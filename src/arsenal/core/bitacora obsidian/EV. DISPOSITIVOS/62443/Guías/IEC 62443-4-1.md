## FR 1: Gestión de la seguridad

### SM-1: Proceso de desarrollo

**PRUEBAS:**
Verificar que los ciclos de vida seguros de desarrollo de productos son efectivos si contienen estos procesos:
*   Gestión de la configuración y del cambio y registros de auditoría
*   Descripción del producto y requisitos con trazabilidad
*   Prácticas de diseño/implementación hardware y software
*   Verificación y validación de pruebas repetibles
*   Revisión y aprobación de registros
*   Soporte del ciclo de vida



### SM-2: Identificación de responsabilidades

**PRUEBAS:**
Verificar que existe un proceso identificativo de roles y responsabilidades para cada proceso de desarrollo, mantenimiento y soporte del producto en el proveedor.



### SM-3: Identificación de la aplicabilidad

**PRUEBAS:**
Verificar que criterios tiene el proveedor del producto para identificar qué productos deben ser desarrollados, mantenidos y apoyados por estos procesos.
Se debe tener en cuenta que estos procesos puedan ser aplicados teniendo en cuenta factores como el mercado destinado para el producto o si se incorpora seguridad o no en este.



### SM-4: Experiencia en seguridad

**PRUEBAS:**
Identificar programas de formación y evaluación en materia de seguridad dirigidos a personal con responsabilidades para adquirir cualificaciones en seguridad y en uso de normas (seguridad, técnicas y herramientas).



### SM-5: Determinación del alcance del proceso

**PRUEBAS:**
Identificar si el proveedor tiene un proceso que justifique mediante un análisis de seguridad si hay procesos que no son aplicables a un proyecto de desarrollo de producto.
Ejemplo: si un producto no tiene software -> procesos relacionados a software no son aplicables.



### SM-6: Integridad del archivo

**PRUEBAS:**
Debe existir un proceso de verificación de integridad para scripts, ejecutables y otros archivos importantes mediante hashes criptográficos y firmas digitales.



### SM-7: Seguridad del entorno de desarrollo

**PRUEBAS:**
Verificar si existen controles técnicos y de procedimiento para la protección del producto, manteniendo la integridad de este durante su desarrollo, producción y entrega.
Protección y actualización durante las fases de diseño, implementación, pruebas y liberación.
Especial atención: protección de autenticadores (contraseñas, ACLs, certificados de firma de código y registros de explotación recopilados en la gestión de defectos).



### SM-8: Controles para claves privadas

**PRUEBAS:**
Verificar que existen controles técnicos y de procedimiento específicos para proteger las claves privadas de firma de códigos para evitar violaciones de la integridad.



### SM-9: Requisitos de seguridad para los componentes suministrados externamente

**PRUEBAS:**
Comprobar que existe un proceso que identifique y gestione los riesgos de seguridad asociados a componentes externos que forman parte de la cadena de suministro del producto, identificando las siguientes características:
*   En qué grado el componente externo se alinea con el contexto de seguridad del producto y la estrategia de defensa en profundidad.
*   Qué grado de rigor se aplica en la aplicación del componente.
*   En qué grado se verifica y valida la seguridad del componente por parte del proveedor del producto o del componente.
*   Cómo se reciben y/o supervisan notificaciones de seguridad del proveedor de componentes.
*   Qué nivel de suficiencia tiene la documentación de seguridad aportada por el proveedor del componente.
*   En qué grado se soporta el software por el proveedor o la comunidad de código abierto.

Se puede satisfacer, por ejemplo, mediante identificación de vulnerabilidades y actualizaciones conocidas de software de código abierto, evaluaciones de conformidad de proveedores de componentes frente a esta norma o similares y mecanismos de compensación por vulnerabilidades.

RECOMENDABLE: que el proveedor del producto tenga un listado de componentes externos para facilitar la gestión de defectos.



### SM-10: Componentes desarrollados a medida por proveedores externos

**PRUEBAS:**
Los componentes de un proveedor tercero deben garantizar que cumplen con los procesos establecidos en este estándar, para así asegurar el ciclo de vida de desarrollo del producto, si:
*   Los componentes se desarrollan específicamente para un único proveedor y para un propósito específico.
*   Los componentes pueden tener algún impacto en la seguridad.



### SM-11: Evaluación y tratamiento de las cuestiones relacionadas con la seguridad

**PRUEBAS:**
Debe existir un proceso que verifique que un producto o parche no es liberado hasta que se resuelvan todos los problemas asociados a la seguridad que superen el riesgo residual aceptable y se haya hecho un seguimiento hasta su cierre, evitando así liberar un producto con problemas de seguridad ya descubiertos o que no hayan sido solventados. Involucra requisitos, seguro por diseño, implementación, verificación, validación y gestión de defectos.



### SM-12: Verificación del proceso

**PRUEBAS:**
Comprobar que el proveedor del producto verifique si los procesos de seguridad exigidos por este estándar se han completado, haciendo uso de registros con documentación de la finalización de cada proceso, garantizando que se lleven a cabo prácticas clave de seguridad.



### SM-13: Mejora continua

**PRUEBAS:**
Debe verificarse la existencia de un proceso que mejore continuamente el ciclo de vida de desarrollo seguro mediante el análisis de defectos de seguridad generales que pueden ir surgiendo por parte de atacantes externos y que escapan del campo.

Ejemplos: utilización de bases de datos de vulnerabilidades, asistencia y realización de conferencias de seguridad, aplicación de análisis y medidas correctivas respecto a vulnerabilidades, herramientas de fuzzing para protocolos sin herramientas disponibles y pruebas de vulnerabilidad para SVV-3 por parte de expertos de seguridad.

## FR 2: Especificación de los requisitos de seguridad

### SR-1: Contexto de seguridad del producto

**PRUEBAS:**
Debe existir documentación acerca del contexto de seguridad del producto, que incluya requisitos mínimos del entorno y las suposiciones sobre el mismo.
Información del contexto de seguridad: ubicación en la red, seguridad física/cibernética del entorno, aislamiento de red e impacto potencial en el entorno (si es conocido).



### SR-2: Modelo de amenaza

**PRUEBAS:**
Todos los productos de un proveedor deben tener un modelo de amenaza específico para el ámbito de desarrollo, revisado y actualizado periódicamente cada año, que contenga:
*   Flujo de información categorizada en todo el sistema.
*   Límites de confianza.
*   Procesos.
*   Almacenamiento de datos.
*   Entidades externas que interactúan.
*   Protocolos de comunicación internos y externos.
*   Puertos físicos accesibles desde el exterior (incluyendo los de depuración).
*   Conexiones de tarjetas de circuito (como JTAG o cabeceras de depuración) que puedan sufrir ataques hardware.
*   Vectores de ataque potenciales (software y hardware).
*   Amenazas potenciales y su gravedad definida por un sistema de puntuación como CVSS o similar.
*   Mitigaciones y/o disposiciones para cada amenaza.
*   Cuestiones relacionadas con la seguridad.
*   Dependencias externas mediante controladores o aplicaciones de terceros.



### SR-3: Requisitos de seguridad del producto

**PRUEBAS:**
El proveedor debe documentar los requisitos de seguridad del producto en desarrollo relacionados con la instalación, funcionamiento, mantenimiento y retirada.

Los requisitos de seguridad del producto aplicados al ciclo de vida del producto son:
*   Privilegios de seguridad para instalar, operar y mantener el producto.
*   Opciones de seguridad, incluyendo eliminación de contraseñas predeterminadas.
*   Consideraciones y acciones de seguridad asociadas a la retirada del producto de su uso.



### SR-4: Contenido de los requisitos de seguridad del producto

**PRUEBAS:**
Asegurar que los requisitos de seguridad incluyan el alcance y límites físicos y lógicos de componentes/sistema y el nivel de seguridad de capacidad requerido del producto.



### SR-5: Revisión de los requisitos de seguridad

**PRUEBAS:**
Verificar que existe un proceso de revisión, actualización y aprobación por parte del proveedor del producto de los requisitos de seguridad. En este proceso deben participar arquitectos/desarrolladores, probadores/testers, defensores del cliente (ventas, marketing, atención al cliente...) y asesores de seguridad.

## FR 3: Seguridad por diseño

Asegurar que el producto es seguro por diseño, incluyendo la defensa en profundidad que proporciona una o más capas de seguridad para frustrar las amenazas de seguridad.
Cada capa de defensa en profundidad está diseñada para proteger los activos de ataques en caso de que todas las demás capas hayan sido comprometidas.
Los procesos requeridos por esta práctica se deben aplicar a todas las etapas del diseño de productos, desde el diseño conceptual hasta el diseño detallado, y todos los niveles del diseño de productos, desde la arquitectura general hasta el diseño de los componentes individuales.

### SD-1: Contexto de seguridad del producto

**PRUEBAS:**
Proceso para desarrollar y documentar el diseño seguro donde se identifique y caracterice cada una de las interfaces del producto, incluidas las interfaces físicas y lógicas, donde se contemple:
1.  Indicación de si la interfaz es accesible externamente (otros productos) o internamente (otros componentes del producto) o ambas cosas.
2.  Implicaciones para la seguridad del contexto de seguridad del producto (ver FR 6) en la interfaz externa.
3.  Los usuarios potenciales de la interfaz y los activos a los que se puede acceder a través de ella (directa o indirectamente).
4.  Determinar si el acceso a la interfaz cruza un límite de confianza.
5.  Consideraciones de seguridad, hipótesis y/o limitaciones respecto al uso de la interfaz en el contexto de la seguridad del producto, incluyendo las posibles amenazas.
6.  Funciones de seguridad, privilegios o derechos y permisos de control de acceso necesarios para utilizar la interfaz y acceder a los activos definidos en el punto (3).
7.  Capacidades de seguridad y/o mecanismos de compensación utilizados para salvaguardar la interfaz y los activos definidos en el punto (3), incluyendo la validación de entradas así como la gestión de salidas y errores.
8.  Uso de productos de terceros para implementar la interfaz y sus capacidades de seguridad.
9.  Documentación que describa como utilizar la interfaz si es accesible desde el exterior.
10. Descripción de como el diseño mitiga las amenazas identificadas en el modelo de amenazas.



### SD-2: Diseño de defensa en profundidad

**PRUEBAS:**
Proceso para implementar múltiples capas de defensa utilizando un enfoque basado en el riesgo y en el modelo de amenaza. Este proceso se debe emplear para asignar responsabilidades a cada capa de defensa.

Verificar que los puertos físicos se encuentran bloqueados, que los puertos lógicos están deshabilitados o limitados, que los puertos o servicios no son accesibles desde el exterior o que se emplean credenciales de acceso.



### SD-3: Revisión del diseño de seguridad

**PRUEBAS:**
Proceso para llevar a cabo revisiones del diseño con el objetivo de identificar, caracterizar y seguir posibles problemas relacionados con la seguridad asociados con cada revisión significativa del diseño seguro incluyendo sin limitarse:
1.  Requisitos de seguridad (FR 6) que no se tuvieran en cuenta en el diseño.
2.  Amenazas y capacidad de explotar las interfaces de los productos, límites de confianza y los activos.
3.  Identificación de prácticas de diseño seguro (SD-4) que no se siguieran.



### SD-4: Mejores prácticas de diseño seguro

**PRUEBAS:**
Proceso para garantizar que se documenten las prácticas de diseño seguro y se apliquen al proceso de diseño.
Las prácticas se deben revisar y actualizar periódicamente, y deben incluir:
1.  Menor privilegio (conociendo solo los privilegios necesarios a los usuarios/software para realizar las operaciones previstas).
2.  Utilización de componentes/diseños de seguridad comprobando siempre que sea posible.
3.  Economía de mecanismo (búsqueda de diseños sencillos).
4.  Uso de patrones de diseño seguros.
5.  Reducción de la superficie de ataque.
6.  Documentar todos los límites de confianza como parte del diseño.
7.  Eliminar los puertos de depuración, las cabeceras y las trazas de las tarjetas de circuito utilizadas durante el desarrollo del hardware, así como documentar su presencia y la necesidad de protegerlos contra el acceso no autorizado.

## FR 4: Implementación segura

### SI-1: Revisión de la implementación de la seguridad

**PRUEBAS:**
El proveedor debe llevar a cabo exámenes de la aplicación con el fin de identificar, caracterizar y seguir las cuestiones relacionadas con la seguridad hasta su cierre.

Estos exámenes deben abordar:
*   Identificación de requisitos de seguridad no abordados correctamente en la aplicación.
*   Identificación de normas de codificación segura que no se siguieron en el desarrollo.
*   Análisis de código estático en el código fuente para encontrar errores de codificación de seguridad (desbordamiento de buffer, punteros nulos, etc.) mediante estándares de codificación seguros y herramientas especializadas.
*   Revisar la implementación y trazabilidad de las capacidades del diseño de seguridad.
*   Examinar las amenazas y su capacidad para explotar interfaces de implementación, límites de confianza y activos.



### SI-2: Normas de codificación segura

**PRUEBAS:**
Se deben implementar normas de codificación de seguridad revisadas y actualizadas periódicamente. Estas normas incluyen:
*   Evitar implementaciones potencialmente explotables.
*   Evitar funciones, patrones de diseño y construcciones de codificación prohibidos.
*   Uso y configuración de herramientas automatizadas.
*   Prácticas de codificación seguras.
*   Validación de entradas que crucen los límites de confianza.
*   Tratamiento de errores.

## FR 5: Verificación de la seguridad y pruebas de validación

### SVV-1: Pruebas de los requisitos de seguridad

**PRUEBAS:**
El proveedor debe verificar mediante un procedimiento que las funciones de seguridad del producto cumplen con los requisitos de seguridad. También se debe comprobar que el producto gestiona correctamente los escenarios de error y las entradas no válidas.

Las pruebas a realizar son:
*   Pruebas funcionales de requisitos de seguridad.
*   Pruebas de rendimiento.
*   Pruebas de escalabilidad.
*   Condiciones de límites y bordes, estrés y pruebas de entradas mal formadas o inesperadas no dirigidas específicamente a la seguridad.



### SVV-2: Pruebas de mitigación de amenazas

**PRUEBAS:**
El proveedor debe establecer un proceso que compruebe si las mitigaciones de amenazas son eficaces y están validadas en el modelo de amenaza. Para ello, es necesario crear y ejecutar planes que aseguren que las mitigaciones han sido probadas adecuadamente (es decir, que funcionan como se tenía previsto en el diseño) y para frustrar cada mitigación (mediante suplantación de identidad, manipulación, denegación de servicio, elevación de privilegios...).



### SVV-3: Pruebas de vulnerabilidad

**PRUEBAS:**
El proveedor debe realizar pruebas referentes a la identificación y caracterización de potenciales vulnerabilidades de seguridad en el producto basadas en fuentes públicas establecidas y reconocidas.

Estas pruebas deben incluir:
*   Pruebas de abuso, entradas mal formadas o inesperadas para descubrir problemas de seguridad, sean manuales o automatizadas e incluyendo pruebas especializadas para interfaces y protocolos.
*   Análisis de la superficie de ataque para determinar entradas y salidas del sistema, vulnerabilidades comunes, puertos expuestos y servicios privilegiados.
*   Realización de pruebas de caja negra.
*   Análisis de composición de software compilado en los binarios ejecutables entregados por el proveedor para su instalación en el producto, que permitan detectar vulnerabilidades conocidas, bibliotecas vulnerables, violaciones de normas de seguridad y configuraciones inseguras del compilador.
*   Pruebas dinámicas de gestión de recursos en tiempo de ejecución que detecten fallos no visibles en el análisis estático del código.



### SVV-4: Ensayo de penetración

**PRUEBAS:**
El proveedor debe realizar pruebas de penetración que identifiquen y caractericen los problemas de seguridad del producto mediante pruebas que descubran y exploten vulnerabilidades.

Algunos ejemplos son eludir autenticación, elevación de privilegios, romper cifrados para comprometer la confidencialidad...



### SVV-5: Independencia de los probadores

**PRUEBAS:**
Debe existir un proceso que garantice que las personas que realizan las pruebas sean independientes de los desarrolladores, siguiendo la siguiente tabla de nivel requerido de independencia.

Los niveles de independencia se dividen en: NINGUNO (no requiere independencia y el propio desarrollador puede hacer las pruebas), PERSONA INDEPENDIENTE (no puede ser desarrollador del producto), DEPARTAMENTO INDEPENDIENTE (debe ser una persona que no informe al mismo gerente de primera línea que un desarrollador) y ORGANIZACIÓN INDEPENDIENTE (persona ajena a la organización desarrolladora del producto).

## FR 6: Gestión de las cuestiones relacionadas con la seguridad

### DM-1: Recepción de notificaciones de cuestiones relacionadas con la seguridad

**PRUEBAS:**
El proveedor debe tener implementado un proceso que permita recibir y dar seguimiento a notificaciones relacionadas a la seguridad del producto.

Estas notificaciones deben provenir de fuentes internas y externas, conformadas por:
*   Probadores de verificación y validación de seguridad.
*   Proveedores de componentes de terceros.
*   Desarrolladores y testers de productos.
*   Usuarios del producto (integradores, propietarios de activos y mantenimiento).

Se incluyen además una serie de directrices a seguir para la presentación de informes de seguridad, las cuales deben ser fácilmente accesibles para las fuentes:
*   Información necesaria para facilitar la validación.
*   Cómo proteger la confidencialidad y el acceso a la información.
*   Grado de comunicación con la entidad que comunica un problema de seguridad.
*   Plazos para informar sobre problemas de seguridad descubiertos internamente.
*   Estrategia para gestionar vulnerabilidades de componentes de terceros descubiertos internamente.



### DM-2: Examen de las cuestiones relacionadas con la seguridad

**PRUEBAS:**
Se debe verificar la existencia de un proceso que garantice que las notificaciones de seguridad se investiguen en un tiempo adecuado para determinar la aplicabilidad, verificabilidad y amenazas que desencadenan el problema.



### DM-3: Evaluación de las cuestiones relacionadas con la seguridad

**PRUEBAS:**
Se debe comprobar que el proveedor realiza análisis de cuestiones de seguridad en el proyecto, en el que se evalúe su impacto respecto al contexto de seguridad real en el que fueron descubiertas, al contexto de seguridad del producto y a la estrategia de defensa en profundidad del producto; la severidad (definida por un sistema de puntuación como CVSS, por ejemplo), la identificación de todos los productos o versiones que contengan la cuestión e identificar las raíces del problema y problemas de seguridad relacionados.



### DM-4: Tratamiento de las cuestiones relacionadas con la seguridad

**PRUEBAS:**
El proveedor del producto debe realizar un tratamiento sobre cuestiones de seguridad para determinar si se deben notificar en base a los resultados de la evaluación de impacto.

Se debe establecer un nivel aceptable de riesgo residual a aplicar al determinar adecuadamente cómo abordar cada cuestión.

El tratamiento de las cuestiones de seguridad se realizará de una o más formas:
*   Arreglar el problema mediante estrategias de defensa en profundidad, cambio de diseño, adición de requisitos y/o capacidades de seguridad, uso de mecanismos de compensación y desactivando o eliminando funciones.
*   Creación de un plan de remediación.
*   Aplazar el problema a futuro y especificar razones y riesgos asociados.
*   No solucionar el problema y asumir el riesgo al ser este menor al nivel aceptable de riesgo residual establecido.

Además, de forma adicional se debe informar a otros procesos del problema o cuestiones conexas e informar a terceros de problemas encontrados en sus códigos fuente.

Debe realizarse una evaluación periódica (cada ciclo de liberación/iteración) de las cuestiones abiertas para garantizar que se abordan correctamente.



### DM-5: Divulgación de cuestiones relacionadas con la seguridad

**PRUEBAS:**
El proveedor debe informar acerca de problemas de seguridad notificables a los usuarios de los productos en tiempo adecuado, informando de:
*   La descripción del problema.
*   Puntuación de vulnerabilidad (CVSS).
*   Versiones de los productos afectados.
*   Desarrollo de la resolución.

Nota: si se trata de una vulnerabilidad asociada a componentes de terceros, se debe tener en cuenta la posibilidad de una divulgación pública prematura por parte del proveedor de componentes de terceros.



### DM-6: Revisión periódica de las prácticas de gestión de defectos de seguridad

**PRUEBAS:**
Debe existir un proceso para realizar revisiones periódicas acerca de la gestión de las cuestiones de seguridad.

Se debe examinar, al menos, las cuestiones ya gestionadas a través del proceso desde el último examen periódico para determinar si el proceso de gestión es eficiente, completo y resolutivo. Esta evaluación es anual.

## FR 7: Gestión de actualizaciones de seguridad

### SUM-1: Calificación de actualización de seguridad

**PRUEBAS:**
Procedimiento que verifique que los parches aplicables se evalúen para garantizar que no afecten negativamente al funcionamiento del producto.
El procedimiento debe verificar:
1.  Actualizaciones de seguridad creadas por el desarrollador contra las vulnerabilidades previstas.
2.  No introduzcan regresiones desarrollando parches creados por:
    1) Desarrollador del producto.
    2) Proveedores de componentes utilizados en el producto.
    3) Proveedores de componentes de los que depende el producto.

El procedimiento debería incluir una configuración que indique que la actualización no contradice otras restricciones operativas, de seguridad o legales.



### SUM-2: Documentación de actualización de seguridad

**PRUEBAS:**
Procedimiento para asegurar que los parches de seguridad estén documentados para permitir la instalación de parches aprobados y la reparación de parches no aprobados.
El procedimiento debe incluir entre otras cosas:
1.  Número de versión del producto al que se aplica el parche de seguridad.
2.  Instrucciones sobre como aplicar los parches aprobados manualmente y mediante proceso automatizado.
3.  Descripción de cualquier impacto que la aplicación del parche al producto pueda tener, incluido el reinicio.
4.  Instrucciones sobre como verificar que se ha aplicado un parche aprobado.
5.  Riesgos de no aplicar el parche y las alternativas que se pueden utilizar para parches que no han sido aprobados o implementados por el propietario del activo.



### SUM-3: Documentación de actualización de seguridad de componentes o sistemas operativos disponibles

**PRUEBAS:**
Proporcionar información a los usuarios sobre si una actualización de seguridad concreta del sistema operativo es compatible con el producto, para ello se debe implementar un proceso contenga:
1.  Indicación de si el producto es compatible con la actualización de seguridad del componente dependiente o del sistema operativo.
2.  Las actualizaciones de seguridad no aprobadas por el proveedor del producto, las mitigaciones que se pueden utilizar en lugar de no aplicar la actualización.



### SUM-4: Entrega de actualizaciones de seguridad

**PRUEBAS:**
Procedimiento para garantizar que los usuarios del producto puedan obtener los parches de seguridad aplicables para el producto de forma oportuna y para reducir la posibilidad de que los parches de seguridad sean fraudulentos, proporcionando un mecanismo o técnica que permita a los usuarios del producto verificar la autenticidad de los parches de seguridad.



### SUM-5: Entrega oportuna de los parches de seguridad

**PRUEBAS:**
Procedimiento que defina una política que especifique los plazos de entrega y calificación de las actualizaciones de seguridad para los usuarios del producto y para garantizar que se sigue esta política, considerando como mínimo los siguientes factores:
1.  Impacto potencial de la vulnerabilidad.
2.  Conocimiento público de la vulnerabilidad.
3.  Existencia de aprovechamientos publicados para la vulnerabilidad.
4.  Volumen de producto desplegados que se ven afectados.
5.  Disponibilidad de una mitigación efectiva en lugar del parche.

## FR 8: Directrices de seguridad

### SG-1: Defensa del producto en profundidad

**PRUEBAS:**
Proceso necesario para asegurar que la documentación para la estrategia de defensa en profundidad se produce para apoyar el fortalecimiento del producto en el emplazamiento del cliente.
Se debe documentar varios aspectos de la estrategia de defensa en profundidad necesaria para fortalecer el producto durante la instalación y mantenerlo fuerte durante su vida útil donde se incluya:
1.  Capacidades de seguridad implementadas por el producto y su papel en la estrategia de defensa en profundidad.
2.  Amenazas abordadas por la estrategia de defensa en profundidad.
3.  Estrategias de mitigación de los usuarios del producto para los riesgos de seguridad conocidos asociados con el producto, incluidos los riesgos asociados con el código heredado.



### SG-2: Medidas de defensa en profundidad esperadas en el entorno

**PRUEBAS:**
Proceso para asegurar que la documentación para la estrategia de defensa en profundidad se produce para apoyar el fortalecimiento del producto en el emplazamiento del cliente.
El proceso debe documentar varios aspectos de la estrategia de defensa en profundidad necesaria para fortalecer el producto durante la instalación y mantenerlo durante su vida útil.



### SG-3: Directrices para el fortalecimiento de la seguridad

**PRUEBAS:**
Proceso para asegurar que las instrucciones que describen como fortalecer el producto y mantenerlo fuerte estén documentadas, donde se incluyan entre otras cosas, instrucciones, fundamentos y recomendaciones para lo siguiente:
1.  Integración del producto.
2.  Integración de las interfaces / protocolos de programación del producto.
3.  Aplicar y mantener la estrategia de defensa en profundidad.
4.  Configuración y uso de opciones/capacidades de seguridad en apoyo de las políticas de seguridad locales y para cada opción/capacidad de seguridad:
    1) Su contribución a la estrategia de defensa en profundidad del producto.
    2) Descripciones de los valores configurables y predeterminados que incluyan la forma en que cada uno afecta a la seguridad, junto con cualquier impacto potencial que cada uno tenga en prácticas de trabajo.
    3) Ajustar / cambiar / borrar su valor.
5.  Instrucciones y recomendaciones para el uso de todas las herramientas y utilidades relacionadas con la seguridad que soportan la administración, supervisión, gestión de incidentes y evaluación de la seguridad del producto.
6.  Instrucciones y recomendaciones para las actividades periódicas de mantenimiento de la seguridad.
7.  Instrucciones para informar de los incidentes de seguridad del producto al proveedor del producto.
8.  Descripción de las mejores prácticas de seguridad para el mantenimiento y la administración del producto.



### SG-4: Directrices para una eliminación segura

**PRUEBAS:**
Proceso necesario para garantizar que se documenten las instrucciones que describen como retirar el producto de manera segura de su uso. El proceso debe incluir instrucciones para eliminar del producto los datos y software sensibles, confidenciales y/o patentados y recomendaciones para lo siguiente:
1.  Retirar el producto de su entorno previsto.
2.  Incluir recomendaciones para eliminar referencias y datos de configuración almacenados en el entorno.
3.  Eliminación segura de los datos almacenados en el producto.
4.  Eliminación segura del producto para evitar la posible divulgación de datos contenidos en el producto que no hayan podido ser eliminados como se describe en el punto (3).



### SG-5: Directrices para una operación segura

**PRUEBAS:**
Proceso para crear la documentación del usuario del producto que describa:
1.  Responsabilidades y acciones necesarias para que los usuarios, incluidos los administradores puedan operar el producto de forma segura.
2.  Supuestos relativos al comportamiento del usuario/administrador y su relación con el funcionamiento seguro del producto.



### SG-6: Directrices para la gestión de cuentas

**PRUEBAS:**
Proceso para crear la documentación del usuario del producto que defina los requisitos de la cuenta del usuario y las recomendaciones asociadas con el uso del producto que incluya:
1.  Permisos de cuentas usuarios (control de acceso) y privilegios (derechos de usuario) necesarios para utilizar el producto, incluidas entre otras, las cuentas del sistema operativo, sistema de control y la base de datos.
2.  Cuentas predeterminadas utilizadas por el producto (cuenta servicio) e instrucciones para cambiar los nombres de cuenta y contraseñas predeterminados.



### SG-7: Revisión de la documentación

**PRUEBAS:**
Procedimiento para identificar, caracterizar y rastrear hasta el cierre los errores y omisiones en todos los manuales de usuario, incluidas las directrices de seguridad que deben incluir:
1.  Cobertura de las capacidades de seguridad del producto.
2.  Integración del producto con el entorno al que está destinado.
3.  Garantía de que todas las prácticas documentadas son seguras.
