# IEC 62443-3-3: Requisitos del Sistema

Lo ideal es que los componentes de un sistema ya tengan la certificación 62443-4-2 para poder centrarnos en la evaluación de la 3-3. No obstante esto suele ocurrir pocas veces. Las pruebas básicas que hay que hacer a cada componente para poder indicar que cumple con los requisitos mínimos son:

> [!INFO] 🚀 Flujo de Trabajo
> **Sigue estos pasos para cada nuevo sistema a auditar:**
> 
> 1. **Crear Bitácora:** Haz clic en el botón de "Nuevo Sistema" justo debajo.
> 2. **Configurar:** Introduce el **Nombre del Dispositivo** y el **Nivel de Seguridad (SL)** objetivo cuando aparezca la ventana emergente. La plantilla se adaptará automáticamente y aparecerá en la carpeta bitácora con la fecha correspondiente.
> 3. **Cumplimentar:** Rellena los campos `Estado::`, `Justificacion::` y `Evidencias::` en cada control de la lista.
> 4. **Visualizar:** Revisa los apartados finales de esta nota (**Dashboard**, **Resumen Detallado** y **Galería de Evidencias**) para ver el porcentaje de cumplimiento y las gráficas actualizadas en tiempo real.

## Checklist de Validación de Seguridad Básica para Componentes (Pre-Auditoría 3-3)

| Comprobación                                               | Cómo Comprobarlo y Herramientas                                                                                                                                                                                                                                                                                   | Nivel de Esfuerzo |
| :--------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------- |
| **FR 1 – Identification and Authentication Control (IAC)** |                                                                                                                                                                                                                                                                                                                   |                   |
| 1. Descubrimiento de Interfaces                            | Realizar un escaneo de puertos completo para identificar todas las interfaces de red (Web, SSH, Telnet, FTP, SNMP, etc.). Herramienta: Nmap. Comando: `nmap -sV -p- <IP_del_componente>`                                                                                                                          | Bajo              |
| 2. Autenticación Obligatoria                               | Intentar acceder a cada interfaz descubierta sin credenciales. El acceso debe ser denegado.                                                                                                                                                                                                                       | Bajo              |
| 3. Ausencia de Credenciales por Defecto                    | Intentar acceder con las credenciales por defecto más comunes para ese tipo de dispositivo (búscalas online) y con listas genéricas (admin/admin, root/root, etc.). Herramienta: Hydra, nmap (con scripts NSE).                                                                                                   | Bajo              |
| 4. Política de Contraseñas                                 | Si tienes acceso de administrador, intenta crear un usuario con una contraseña débil (ej. "1234"). El sistema debería rechazarla. Revisa si la política de complejidad es configurable.                                                                                                                           | Medio             |
| 5. Protección contra Fuerza Bruta                          | Realizar múltiples intentos de login fallidos con una cuenta de prueba. La cuenta debería bloquearse temporalmente. Herramienta: Hydra, Burp Suite Intruder.                                                                                                                                                      | Medio             |
| 6. Transmisión Segura de Credenciales                      | Capturar el tráfico de red durante un inicio de sesión. Verificar que las credenciales no viajan en texto plano. El login web debe usar HTTPS, no HTTP. Herramienta: Wireshark.                                                                                                                                   | Bajo              |
| 7. Almacenamiento Seguro de Credenciales                   | (Avanzado) Si puedes obtener el firmware o acceso al sistema de ficheros, busca ficheros que almacenen contraseñas (ej. /etc/shadow, ficheros de configuración). Verifica que las contraseñas están hasheadas con un algoritmo robusto. Herramientas: Binwalk, strings, grep.                                     | Alto              |
| **FR 2 – Use Control (UC)**                                |                                                                                                                                                                                                                                                                                                                   |                   |
| 1. Existencia de Roles y Privilegios                       | Verificar en la interfaz de gestión si existen diferentes niveles de usuario (ej. Administrador, Operador, Solo Lectura).                                                                                                                                                                                         | Bajo              |
| 2. Separación de Privilegios                               | Inicia sesión con una cuenta de bajos privilegios (ej. "Solo Lectura"). Intenta realizar acciones administrativas (cambiar configuración, crear usuarios, reiniciar). Todas las acciones deben ser denegadas. Herramienta: Burp Suite (para intentar forzar peticiones de admin con una cookie de usuario).       | Medio             |
| **FR 3 – System Integrity (SI)**                           |                                                                                                                                                                                                                                                                                                                   |                   |
| 1. Verificación de Integridad del Firmware                 | Descarga una actualización de firmware oficial. Modifica un solo bit del fichero. Intenta subir el fichero modificado. El dispositivo debe rechazarlo con un error de checksum o integridad. Herramienta: Un editor hexadecimal como HxD.                                                                         | Medio             |
| 2. Verificación de Autenticidad del Firmware               | (Avanzado) Intenta firmar un fichero de firmware (el original o el modificado) con tu propio certificado autofirmado. Intenta subirlo. El dispositivo debe rechazarlo por "firma no válida" o "firmante no confiable". Herramienta: OpenSSL.                                                                      | Alto              |
| 3. Validación de Entradas (Input Validation)               | En todas las interfaces web y de red, introduce datos malformados. Busca vulnerabilidades básicas de inyección: <br> - XSS: `<s>alert(1)</s>` en campos de texto. <br> - Inyección de Comandos: `127.0.0.1; ls -la` en campos de IP. <br> - SQLi: `' OR 1=1 --` en campos de login. <br> Herramienta: Burp Suite. | Medio             |
| **FR 4 – Data Confidentiality (DC)**                       |                                                                                                                                                                                                                                                                                                                   |                   |
| 1. Cifrado de Datos en Tránsito                            | Captura el tráfico de red durante la operación normal. Verifica que la información de configuración o de proceso sensible no viaja en texto plano. Herramienta: Wireshark.                                                                                                                                        | Bajo              |
| 2. Cifrado de Datos en Reposo                              | Si puedes exportar una configuración, ábrela y comprueba si contiene contraseñas o claves en texto plano.                                                                                                                                                                                                         | Bajo              |
| **FR 7 – Resource Availability (RA)**                      |                                                                                                                                                                                                                                                                                                                   |                   |
| 1. Resiliencia a Denegación de Servicio (DoS)              | Lanza un ataque de bajo volumen contra una interfaz de gestión (ej. la interfaz web). Mientras el ataque está en curso, verifica si la función principal del dispositivo sigue operativa (ej. si es un PLC, que sigue ejecutando su lógica). Herramientas: hping3, slowhttptest.                                  | Medio             |
| 2. Manejo de Errores                                       | Envía peticiones malformadas o basura al dispositivo. El dispositivo debe manejar el error de forma controlada y no debe reiniciarse o bloquearse. Los mensajes de error no deben filtrar información sensible (stack traces, rutas internas, etc.). Herramienta: Burp Suite Repeater/Fuzzer.                     | Medio             |

## Definición SuC / Zonas / Untrusted network

**Guía Práctica para el Taller de Alcance y Segmentación**

Este taller es una reunión (o serie de reuniones) cuyo único objetivo es definir y acordar el alcance de la auditoría.

**Fase 1: Preparación (Antes de la reunión)**
No vayas a la reunión en blanco. Tu preparación demuestra profesionalidad y dirige la conversación.
1.  **Identifica a los Asistentes Clave:** Necesitas a la gente que conoce el sistema, no solo a los directivos. Invita a:
    *   Responsable de Operaciones/Jefe de Planta: Conoce el proceso industrial, la criticidad y el impacto de una parada. Es el dueño del riesgo.
    *   Ingeniero de Automatización/Control: Conoce los PLCs, el SCADA, los protocolos y cómo funciona el sistema a bajo nivel.
    *   Administrador de la Red OT: Conoce los switches, firewalls y la arquitectura de la red de control.
    *   Administrador de la Red IT: Clave para definir la frontera y los controles en el conduit que conecta IT y OT.
    *   Responsable de Ciberseguridad (CISO o similar): Aporta la visión de la política de seguridad global de la empresa.
2.  **Solicita Documentación Previa:** Pide que te envíen con antelación:
    *   Diagramas de Arquitectura de Red: Es el documento más importante.
    *   Inventario de Activos (si existe): Lista de servidores, PLCs, switches, etc.
    *   (Consejo de experto) Diagramas de Proceso (P&ID): Te ayuda a entender la función industrial del sistema y, por tanto, su criticidad. Te permite hacer preguntas como "¿Qué pasa si este PLC falla?".

**Fase 2: El Taller (Dirigiendo la reunión)**
Tu objetivo es salir de aquí con un borrador del mapa de Zonas y Conduits. Usa una pizarra blanca, es tu mejor herramienta.

*   **Paso 1: Definir el SuC (System under Consideration)**
    *   → Pregunta Clave: "¿Qué sistema vamos a auditar exactamente?"
    *   Acción: Empieza por el nombre funcional. No digas "la red de PLCs". Di "el sistema de control de la línea de embotellado nº 3" o "el sistema de gestión de la caldera B". Debe ser algo que el Jefe de Planta entienda perfectamente. Dibuja un gran recuadro en la pizarra y ponle ese nombre. Todo lo que dibujemos dentro será el SuC.

*   **Paso 2: Dibujar el Mapa de Activos (Visión General)**
    *   Pregunta Clave: "Basándonos en los diagramas, ¿cuáles son los componentes principales de este sistema?"
    *   Acción: Dibuja los componentes principales dentro del recuadro del SuC: los servidores SCADA, los HMIs, los PLCs, el historiador, los switches principales. No entres en detalles todavía.

*   **Paso 3: Identificar las Zonas (Agrupar por Función y Criticidad) "Las salas donde se encuentran estos componentes"**
    *   Pregunta Clave: "Ahora, agrupemos estos dispositivos. ¿Cuáles tienen la misma función y necesitan el mismo nivel de protección?"
    *   Acción: Dibuja círculos o recuadros alrededor de los grupos de activos. Guía la conversación con estas preguntas:
        *   Por Función: "¿Estos son todos PLCs de control de proceso? OK, Zona de Control. ¿Estos son los servidores que los operadores miran? OK, Zona de Supervisión."
        *   Por Criticidad: "¿Tenemos algún Sistema Instrumentado de Seguridad (SIS)? Ese debe ir en su propia zona, la Zona de Seguridad, porque sus requisitos son mucho más altos."
        *   Por Confianza: "¿Dónde está el servidor al que se conectan los técnicos de mantenimiento desde fuera? OK, esa es la DMZ Industrial."

*   **Paso 4: Identificar los Conduits (Las Conexiones) "Las Puertas que permiten acceder a las Salas"**
    *   Pregunta Clave: "Perfecto, ya tenemos las zonas. Ahora, ¿cómo se comunican entre ellas?"
    *   Acción: Dibuja las líneas que conectan tus zonas. Para cada línea, pregunta:
        *   "¿Qué dispositivo físico controla esta comunicación? ¿Un firewall? ¿Un router con ACLs?" Ese dispositivo es el conduit.
        *   "¿Qué protocolos y puertos necesitan cruzar por aquí? ¿Solo Modbus en el puerto 502 desde este HMI a este PLC?" Esto será crucial para la auditoría del SR 5.2.

*   **Paso 5: Identificar las Redes No Confiables (Untrusted Networks)**
    *   Pregunta Clave: "¿Qué hay fuera de nuestro SuC? ¿Con qué se conecta este sistema?"
    *   Acción: Dibuja las nubes o recuadros externos. Las más comunes son:
        *   La Red Corporativa (IT): Desde la perspectiva de OT, la red IT es no confiable por defecto. El conduit que la conecta es el firewall perimetral IT/OT.
        *   Internet: Si hay alguna conexión directa (¡muy mala práctica!).
        *   Redes de Terceros: La conexión VPN que usa el proveedor del PLC para mantenimiento es un conduit hacia una red no confiable.

**Fase 3: Documentación (El Entregable)**
*   Acción: Saca una foto de la pizarra. Pasa ese dibujo a una herramienta como Visio o draw.io.
*   Entregable: Crea un documento de "Definición de Alcance" que incluya:
    *   La descripción del SuC.
    *   El diagrama de Zonas y Conduits.
    *   Una tabla que liste los activos principales dentro de cada zona.
    *   Una tabla que describa cada conduit, el dispositivo que lo protege y los flujos de datos permitidos.
*   Acuerdo Final: Envía este documento a todos los asistentes y pide su confirmación explícita. Este documento es tu contrato de auditoría. Cualquier cosa que no esté aquí, está fuera de alcance.

> 💡 **Nota de Experto: "El Conducto Humano"**
> No te olvides del "conduit humano". Pregunta siempre: "¿Cómo se actualiza y mantiene este sistema?". La respuesta casi siempre es: "Un ingeniero se conecta con su portátil".
> Ese portátil es un conduit peligrosísimo. Se conecta a la red IT (para recibir emails, navegar) y luego a la red OT (para programar el PLC). Es un puente directo.
> Por tanto, la seguridad de ese portátil (hardening, antivirus, control de USB, etc.) debe formar parte del alcance de tu auditoría 3-3, aunque el portátil en sí no sea un componente fijo del IACS.

    *   Verificación de Logs de Acceso: Revisar los logs del sistema para confirmar que todos los intentos de conexión (exitosos y fallidos) desde la red no confiable son registrados.
    *   [Herramientas]: Nmap, hping3.

*   **SL3/4 (Añade RE 1: Aprobación explícita):**
    *   Prueba de Denegación por Defecto: Intentar conectar desde una red no confiable sin ninguna regla de permiso explícita. La conexión debe ser rechazada por defecto.
    *   Prueba de Autorización Basada en Roles: Asignar a un usuario un rol que permita el acceso remoto y verificar que la conexión es exitosa. Revocar el rol y verificar que la conexión es denegada.
    *   [Herramientas]: Cliente de red (SSH, HTTPS), interfaz de gestión.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Nmap (Escaneo externo):** `nmap -Pn -sS <IP>` (Desde una subred distinta).
> *   **Hping3 (Prueba de ACL):** `hping3 -S -p 80 -c 1 <IP>` (Verificar si llega SYN-ACK o nada).
